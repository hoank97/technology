# 6.2 ACID vs BASE

## CAP Theorem — Nền tảng lý thuyết

**Eric Brewer's CAP Theorem** (2000): trong một distributed system, chỉ có thể đảm bảo đồng thời **2 trong 3** properties:

```
CAP Triangle

        Consistency (C)
           /    \
          /      \
         /        \
        /          \
    Availability (A) —— Partition Tolerance (P)

C + A: Không có partition → đảm bảo cả C và A
C + P: Partition xảy ra → phải chọn C hoặc A
A + P: Chấp nhận eventual consistency (Base)
```

**Định nghĩa 3 properties:**

| Property | Ý nghĩa |
|----------|---------|
| **Consistency** | Mọi read đều nhận được write gần nhất hoặc lỗi |
| **Availability** | Mọi request đều nhận được response (có thể không phải latest) |
| **Partition Tolerance** | Hệ thống tiếp tục hoạt động khi network partition xảy ra |

> ⚠️ **Partition xảy ra** = network failure chia hệ thống thành 2+ phần không thể communicate. Đây là **inevitable** trong distributed systems — bạn không thể chọn "không có partition". Vì vậy thực tế chỉ có 2 lựa chọn: **CP** hoặc **AP**.

## ACID

### Định nghĩa

**ACID** = Atomicity, Consistency, Isolation, Durability — là model của **traditional relational databases** (PostgreSQL, MySQL, Oracle).

**Đảm bảo strong consistency**: mọi transaction đều có kết quả có thể dự đoán được. Khi commit thành công → data được ghi bền vững, mọi reader đều thấy giá trị mới.

### 4 properties

| Property | Ý nghĩa | Ví dụ |
|----------|---------|-------|
| **Atomicity** | Transaction là "all or nothing" — hoặc tất cả thay đổi được apply, hoặc không có gì | Transfer: debit + credit hoặc không có gì |
| **Consistency** | Database luôn ở valid state — không vi phạm constraints | Balance không âm, FK luôn hợp lệ |
| **Isolation** | Các concurrent transactions không interfere lẫn nhau | T2 không thấy T1's uncommitted changes |
| **Durability** | Commit thành công → survive system crash | WAL đảm bảo data được persist |

### Isolation Levels (ANSI SQL)

Để balance giữa **correctness** và **performance**, có 4 isolation levels:

| Level | Dirty Read | Non-repeatable Read | Phantom Read |
|-------|------------|---------------------|-------------|
| **READ UNCOMMITTED** | ✅ Có thể | ✅ Có thể | ✅ Có thể |
| **READ COMMITTED** | ❌ Không | ✅ Có thể | ✅ Có thể |
| **REPEATABLE READ** | ❌ Không | ❌ Không | ⚠️ PostgreSQL: không có |
| **SERIALIZABLE** | ❌ Không | ❌ Không | ❌ Không |

```sql
-- PostgreSQL: kiểm tra isolation level hiện tại
SHOW transaction_isolation;

-- Set isolation level
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- Sử dụng trong practice:
-- SERIALIZABLE: financial transactions (tiền)
-- READ COMMITTED: most business logic (default PostgreSQL)
-- READ UNCOMMITTED: almost never
```

```go
// Go: thiết lập isolation level
tx, _ := db.BeginTx(ctx, &sql.TxOptions{
    Isolation: sql.LevelSerializable,
})
```

### Trade-offs

| | ✅ Pros | ⚠️ Cons |
|--|--|--|
| **Consistency** | Data never in invalid state | Wait for confirmation = slower writes |
| **Isolation** | Predictable concurrent behavior | Lock contention = throughput limit |
| **Durability** | Survive crashes, no data loss | fsync = I/O overhead |

## BASE

### Định nghĩa

**BASE** = **Basically Available, Soft state, Eventually consistent** — model của **NoSQL** distributed databases (Cassandra, DynamoDB, Riak).

**Chấp nhận eventual consistency**: không đảm bảo data nhất quán ngay lập tức, nhưng hệ thống vẫn **available** trong mọi tình huống.

```
ACID: Strong consistency + Available khi không có partition
BASE: Available always + Eventual consistency (có thể đợi)
```

### 3 properties

| Property | Ý nghĩa |
|----------|---------|
| **Basically Available** | System đảm bảo availability (hơn là guarantee consistency) |
| **Soft state** | State có thể thay đổi theo thời gian (vì eventual consistency) |
| **Eventually consistent** | Nếu không có new updates, cuối cùng tất cả replicas sẽ converge |

### Eventual Consistency types

```
Strong → Sequential → Causal → Eventual (weakest)

Strong:        Đọc ngay giá trị mới nhất
Sequential:    Tất cả readers thấy cùng order của writes
Causal:        Causally-related operations thấy đúng order
Eventual:      Có thể đọc giá trị cũ trong một thời gian
```

**Tunable consistency** (Cassandra, DynamoDB):

```sql
-- Cassandra: consistency levels
CONSISTENCY ONE      -- 1 node acknowledge = fast, weak
CONSISTENCY QUORUM   -- >50% nodes = balance
CONSISTENCY ALL      -- Tất cả replicas = strong consistency
CONSISTENCY EACH_QUORUM -- Mỗi datacenter >50%
```

```javascript
// DynamoDB: read consistency
// Strongly consistent read (read-after-write)
dynamodb.getItem({ TableName: 'Users', Key: {...}, ConsistentRead: true })

// Eventual consistent read (default, faster)
dynamodb.getItem({ TableName: 'Users', Key: {...}, ConsistentRead: false })
```

### Vector Clocks — Ghi nhớ causality

```javascript
// Vector clock ghi nhớ causality của mỗi update
// Thay vì timestamp đơn giản (không reliable trong distributed)
// Vector clock track version history của từng node

// Node A writes: { A: 2, B: 1 } → version mới
// Node B writes: { A: 1, B: 3 } → version mới

// Để detect conflict: so sánh vector clocks
// { A: 2, B: 1 } vs { A: 1, B: 3 } → incomparable → CONFLICT!
// Cassandra dùng timestamp đơn giản hơn (last-write-wins)
```

## ACID vs BASE — So sánh toàn diện

### Conceptual differences

| Aspect | ACID (SQL) | BASE (NoSQL) |
|--------|------------|--------------|
| **Focus** | Correctness trước | Availability trước |
| **Consistency** | Strong (immediate) | Eventual (async) |
| **Model** | Pessimistic locking | Optimistic, conflict resolution |
| **Latency** | Higher (waits for confirmation) | Lower (fire-and-forget) |
| **Network failure** | Fail & return error | Continue with stale data |
| **Data loss on crash** | Minimal | Possible (depends on config) |
| **Typical use cases** | Financial, inventory, orders | Activity logs, sessions, catalogs |

### Performance characteristics

| | ACID (PostgreSQL) | BASE (Cassandra) | BASE (DynamoDB) |
|--|--|--|--|
| **Write latency** | ~5-20ms (fsync) | ~1-5ms (no fsync) | ~10-30ms |
| **Read latency** | ~1-5ms | ~1-5ms | ~5-15ms |
| **Throughput** | Limited by locking | High (no locks) | Scales linearly |
| **Conflict resolution** | DB-level lock | Last-write-wins / custom | Client-side merge |

### Real-world decision matrix

| Scenario | ACID? | BASE? | Why |
|----------|-------|-------|-----|
| **Bank transfer** | ✅ | ❌ | Money must never be lost |
| **Inventory stock** | ✅ | ⚠️ | Stock mismatch = oversell |
| **User sessions** | ❌ | ✅ | TTL-based, short-lived |
| **Product catalog** | ⚠️ | ✅ | Stale reads OK, fast reads critical |
| **Chat messages** | ❌ | ✅ | Eventual = acceptable |
| **Payment ledger** | ✅ | ❌ | Audit trail must be exact |
| **Leaderboard** | ❌ | ✅ | Periodic sync OK |
| **Order history** | ⚠️ | ✅ | Eventually consistent is fine |

## ⚠️ Hybrid: ACID + BASE trong production

### PostgreSQL — cũng có eventual consistency

```sql
-- Read replicas (streaming replication)
-- Primary: strong consistency (writes)
-- Replica: eventual consistency (reads, lag ~ms)

-- App chọn consistency level
SET TRANSACTION READ ONLY;  -- READ COMMITTED on replica

-- Asynchronous commit (trading durability for speed)
ALTER TABLE orders SET (sync_method = 'async');
```

### Cassandra — cũng có tunable strong consistency

```sql
-- Cross-DC replication với consistency ALL = strong
CONSISTENCY ALL;  -- Mọi replicas phải ack = strong consistency

-- Trade-off: nếu 1 node down → request fail
CONSISTENCY QUORUM;  -- >50% = strong enough, tolerate some failures
```

### Eventual consistency patterns

```go
// 1. Read-your-writes consistency (eventual but fast)
type UserService struct {
    db        *sql.DB  // primary (strong)
    readDB    *sql.DB  // replica (eventual)
}

func (s *UserService) UpdateEmail(ctx context.Context, userID, email string) error {
    // Write to primary (strong consistency)
    _, err := s.db.ExecContext(ctx, "UPDATE users SET email=$1 WHERE id=$2", email, userID)
    return err
}

func (s *UserService) GetUser(ctx context.Context, userID string) (*User, error) {
    // Read from primary (not replica) for this user
    // → đảm bảo read-your-writes
    return s.getFromPrimary(ctx, userID)
}
```

```go
// 2. Conflict resolution: Last-Write-Wins
// Cassandra: timestamp-based, simple nhưng có thể lose updates

// 3. Conflict resolution: Custom (vector clock + application logic)
type Order struct {
    ID        string
    Status    string
    VectorClock map[string]int64  // track causality
}

func (o *Order) Merge(incoming Order) Order {
    // Application decides how to resolve conflict
    if incoming.VectorClock["node_A"] > o.VectorClock["node_A"] {
        o.Status = incoming.Status  // accept incoming
    }
    // ... merge logic
    return o
}
```

## 📌 Tóm tắt

```
ACID vs BASE
├── CAP Theorem
│   ├── C + A + P: chỉ chọn 2 trong 3
│   ├── P là inevitable → chỉ còn CP hoặc AP
│   └── ACID = CP, BASE = AP
│
├── ACID (Relational DB)
│   ├── Atomicity: all-or-nothing
│   ├── Consistency: valid state
│   ├── Isolation: no interference
│   ├── Durability: survive crash
│   ├── ✅ Strong consistency, correct data
│   ├── ⚠️ Higher latency, lock contention
│   └── Use: financial, inventory, orders
│
├── BASE (NoSQL)
│   ├── Basically Available: always serving
│   ├── Soft state: may change
│   ├── Eventually consistent: converge over time
│   ├── ✅ High availability, low latency, scale out
│   ├── ⚠️ May read stale data, conflicts need resolution
│   └── Use: sessions, logs, catalogs, social data
│
└── Hybrid in production
    ├── PostgreSQL: primary (strong) + replicas (eventual)
    ├── Cassandra: tunable consistency QUORUM/ALL
    └── App patterns: read-your-writes, conflict resolution
```
