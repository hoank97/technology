# 6.1 SQL vs NoSQL

## SQL — Relational Databases

### Định nghĩa

**SQL databases** (Relational DB) tổ chức dữ liệu trong **tables với fixed schema**. Mỗi row là một record, mỗi column có kiểu định nghĩa trước. Quan hệ giữa các bảng được biểu diễn qua **foreign keys** và truy vấn qua **JOINs**.

**Storage engine**: Dùng **B-Tree** (hoặc variant) để lưu trữ data trên disk. Index thường là clustered hoặc secondary B-Tree indexes. Data được tổ chức theo pages (thường 8KB/16KB), mỗi page chứa rows.

```sql
-- PostgreSQL example: normalized schema
CREATE TABLE users (
    id    BIGSERIAL PRIMARY KEY,
    email VARCHAR(255) UNIQUE NOT NULL,
    name  VARCHAR(255)
);

CREATE TABLE orders (
    id         BIGSERIAL PRIMARY KEY,
    user_id    BIGINT REFERENCES users(id),
    total      DECIMAL(10, 2),
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- JOIN — SQL's superpower
SELECT u.email, COUNT(o.id) AS order_count, SUM(o.total) AS total_spent
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
WHERE u.created_at > '2024-01-01'
GROUP BY u.id, u.email
HAVING SUM(o.total) > 100
ORDER BY total_spent DESC;
```

**Scaling behavior**: SQL truyền thống scale **vertically** (thêm CPU/RAM vào một máy). Với partitioning/sharding có thể scale horizontally, nhưng JOINs trở nên phức tạp hơn nhiều khi data nằm trên nhiều nodes.

## NoSQL — Not Only SQL

### Định nghĩa

**NoSQL databases** là một nhóm rộng gồm nhiều loại, mỗi loại tối ưu cho một use case khác nhau:

| Loại | Ví dụ | Đặc điểm |
|------|-------|-----------|
| **Document** | MongoDB, Couchbase | JSON-like documents, flexible schema |
| **Key-Value** | Redis, DynamoDB | O(1) lookup, đơn giản |
| **Wide-Column** | Cassandra, HBase | Tốt cho time-series, log data |
| **Graph** | Neo4j | Xử lý relationships chuyên sâu |
| **Search** | Elasticsearch | Full-text search, analytics |

```javascript
// MongoDB: document store — flexible schema, nested data
{
  _id: ObjectId("..."),
  user_id: 12345,
  email: "user@example.com",
  profile: {
    first_name: "Alice",
    last_name: "Tran",
    preferences: {
      theme: "dark",
      language: "vi"
    }
  },
  orders: [
    { item: "Laptop", amount: 15000000, date: "2025-03-15" },
    { item: "Mouse", amount: 450000, date: "2025-04-01" }
  ],
  tags: ["premium", "vip"]
}
// → Schema không cố định, có thể thêm field mới mà không ALTER TABLE
```

**Storage engine**: Khác nhau tùy loại:
- **Document DB**: B-Tree hoặc LSM-Tree (WiredTiger in MongoDB)
- **Key-Value**: In-memory hash map, có thể persisted qua Write-Ahead Log + snapshot
- **Wide-Column**: Sorted String Table (SSTable) format, tối ưu cho append-only workload

**Scaling behavior**: NoSQL được thiết kế cho **horizontal scaling** từ đầu. Cassandra dùng consistent hashing để partition data. MongoDB dùng shard keys. Redis Cluster chia data theo hash slots.

## So sánh chi tiết

### Data Model

| | SQL (PostgreSQL) | NoSQL (MongoDB) | NoSQL (Redis) |
|--|--|--|--|
| **Model** | Tables + Rows | Collections + Documents | Key-Value |
| **Schema** | Fixed (migrations needed) | Flexible (dynamic fields) | Schema-less |
| **Relationships** | Foreign keys + JOINs | Embedded or referenced documents | Denormalized |
| **Nesting** | Normalize → separate tables | Embed directly in document | Store as serialized value |
| **Transactions** | ACID (multi-document) | ACID (single collection) | Single-key atomic |

### Consistency & Scaling

| | SQL (PostgreSQL) | NoSQL (Cassandra) | NoSQL (MongoDB) |
|--|--|--|--|
| **Default consistency** | Strong (read-your-writes) | Eventual (tunable) | Strong (single-node), Eventual (replica set) |
| **Scaling** | Vertical + limited horizontal | Horizontal (native) | Horizontal (sharding) |
| **Join support** | ✅ Full SQL JOINs | ❌ No joins (denormalize) | ❌ No joins |
| **ACID transactions** | ✅ Multi-document, cross-table | ⚠️ Lightweight transactions (tunable) | ✅ Lua scripts + transactions |
| **Hot data** | Query cache | Integrated cache (WiredTiger) | Built-in (in-memory) |
| **Write throughput** | ~10K-50K ops/s (single node) | ~100K+ ops/s (cluster) | ~100K-1M ops/s (tùy data size) |

### Operational complexity

| | SQL | NoSQL |
|--|--|--|
| **Setup** | Straightforward | Varied (Cassandra cluster = complex) |
| **Schema migrations** | Mature tools (Flyway, Liquibase) | ❌ No schema — code handles it |
| **Query language** | Standardized SQL | Proprietary APIs (mongosh, CQL) |
| **Backup/restore** | Mature (pg_dump, PITR) | DB-specific (varies widely) |
| **Monitoring** | Mature tooling (pg_stat_statements, EXPLAIN ANALYZE) | Varies by DB |

## Khi nào chọn cái nào

### Chọn SQL khi:

| ✅ Use case | Lý do |
|---|---|
| **Financial transactions** | ACID guarantees, không mất tiền |
| **Complex relational data** | Orders → Order Items → Products → Vendors — JOINs là thế mạnh |
| **Reporting & analytics** | Complex aggregations, window functions |
| **Data integrity là số 1** | Foreign keys, constraints, unique indexes |
| **Team đã có SQL skill** | Ít learning curve |

### Chọn NoSQL khi:

| ✅ Use case | Loại NoSQL |
|---|---|
| **User profiles, catalogs** | Document (MongoDB) |
| **Sessions, cache, leaderboard** | Key-Value (Redis) |
| **IoT sensor data, logs** | Wide-Column (Cassandra) |
| **Recommendations, social graph** | Graph (Neo4j) |
| **Search + analytics dashboard** | Search (Elasticsearch) |
| **Schemas thay đổi liên tục** | Document |

## ⚠️ Common pitfalls

### 1. SQL — Over-normalizing
```sql
-- ❌ Extreme normalization: join hell
SELECT * FROM orders o
JOIN order_items oi ON oi.order_id = o.id
JOIN products p ON p.id = oi.product_id
JOIN vendors v ON v.id = p.vendor_id
JOIN vendor_addresses va ON va.vendor_id = v.id
... -- 8+ joins for a simple query

-- ✅ Denormalize when read-heavy
SELECT * FROM order_summary WHERE user_id = $1;
```

### 2. NoSQL — Missing index on shard key
```javascript
// ❌ Query cross-shards = full cluster scan
db.orders.find({ created_at: { $gte: ISODate("2025-01-01") } })

// ✅ Filter by shard key = targeted
db.orders.find({ user_id: "user_123", created_at: { $gte: ISODate("2025-01-01") } })
```

### 3. NoSQL — Embedding too much
```javascript
// ❌ Embed everything = document bloat, hot spot
{
  user_id: "vip_user",
  entire_order_history: [/* 5000 orders */] // 50MB document!
}

// ✅ Reference when unbounded array
// orders collection: { user_id, items, ... }
// users collection: { user_id, profile, ... }
```

## 📌 Tóm tắt

```
SQL vs NoSQL
├── SQL (Relational)
│   ├── ✅ ACID guarantees, complex JOINs, strong consistency
│   ├── ✅ Mature tooling, standard SQL, data integrity
│   ├── ⚠️ Vertical scaling, fixed schema, migrations complexity
│   └── Use when: financial, reporting, structured relational data
│
├── NoSQL types
│   ├── Document (MongoDB): flexible schema, embedded data
│   ├── Key-Value (Redis): O(1), sessions, cache
│   ├── Wide-Column (Cassandra): high write throughput, time-series
│   ├── Graph (Neo4j): deep relationships
│   └── Search (Elasticsearch): full-text, analytics
│
└── Decision tree
    ├── Need ACID + complex queries? → SQL
    ├── Schema changes often? → Document NoSQL
    ├── Need horizontal scale + high writes? → Cassandra/DynamoDB
    └── Need extreme speed + caching? → Redis
```
