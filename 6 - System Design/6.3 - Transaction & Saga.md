## Distributed Transaction

Transaction cross nhiều databases/services.

### Problem

```
Service A (DB1)              Service B (DB2)
     │                             │
     │  Update inventory ✓         │
     │───────────────────────────  │
     │                             │  Update payment ✗ (timeout)
     │                             │  → Rollback inventory? → KHÔNG CÓ CÁCH!
     │                             │
     ← Cannot rollback! ←
```

→ **2-Phase Commit (2PC)** giải quyết nhưng phức tạp.

---

## Saga Pattern

A **sequence of local transactions**. If a step fails → execute **compensating transactions** to undo previous steps.

> Không có 2PC → dùng **compensations** thay vì rollback.

### Example: Order + Payment + Inventory

```
Step 1: CreateOrder    → Compensate: CancelOrder
Step 2: ReserveInventory → Compensate: ReleaseInventory
Step 3: ChargePayment   → Compensate: RefundPayment
Step 4: ConfirmOrder    → Compensate: CancelConfirmation
```

---

## Approaches

### 1. Choreography (Event-driven, Decentralized)

```
Order Service ──→ Payment Service ──→ Inventory Service
      ↑                   │
      └───────────────────┘
         (events trigger next steps)
```

**Tốt cho**: Simple workflows, few services
**Xấu cho**: Complex coordination, many services

### 2. Orchestration (Centralized Coordinator)

```
Saga Orchestrator
    │
    ├── Step 1: CreateOrder
    ├── Step 2: ReserveInventory   (rollback: ReleaseInventory)
    ├── Step 3: ChargePayment      (rollback: RefundPayment)
    └── Step 4: ConfirmOrder
```

**Tốt cho**: Complex workflows, many services
**Xấu cho**: Simple workflows (overkill)

## Implementation

```go
type OrderSaga struct {
    orderSvc    OrderService
    paymentSvc  PaymentService
    inventorySvc InventoryService
}

func (s *OrderSaga) Execute(ctx context.Context, req CreateOrderReq) error {
    order, err := s.orderSvc.Create(ctx, req)
    if err != nil {
        return err
    }

    if err := s.inventorySvc.Reserve(ctx, order.ID); err != nil {
        s.orderSvc.Cancel(ctx, order.ID) // compensate
        return err
    }

    if err := s.paymentSvc.Charge(ctx, order.ID, req.Amount); err != nil {
        s.inventorySvc.Release(ctx, order.ID) // compensate
        s.orderSvc.Cancel(ctx, order.ID)     // compensate
        return err
    }

    return s.orderSvc.Confirm(ctx, order.ID)
}
```

## Outbox Pattern

Tránh **lost event** khi DB write success nhưng MQ publish fail.

```
Without Outbox:
1. DB write ✓ → 2. MQ publish ✗ → Event LOST!

With Outbox:
1. DB write + outbox record ✓ (atomic, cùng transaction)
2. Outbox poller đọc → 3. MQ publish ✓ → 4. Mark sent ✓
```

→ **At-least-once delivery**, không mất event.
