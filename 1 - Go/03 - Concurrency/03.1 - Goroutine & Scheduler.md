# 03.1 Goroutine & Scheduler

## Go Scheduler (G-M-P Model)

### Components

| | Mô tả |
|---|---|
| **G** (Goroutine) | Goroutine đang chạy, chứa stack & instruction pointer |
| **M** (Machine) | OS thread thực sự |
| **P** (Processor) | Resource để schedule G (số lượng = GOMAXPROCS) |

### Rules

1. G chạy trên M
2. M cần P để execute Go code
3. GOMAXPROCS = số lượng P

```go
runtime.GOMAXPROCS(4) // Enable parallelism với 4 CPU cores
```

---

### Missions

#### Multiplexing

Reflex N goroutines → M OS threads → P processors

#### Work Stealing

Khi P hết việc → steal G từ P khác → **Load balance**

#### Preemption

Preemption là ability của Go runtime để **pause một running goroutine** để others can run.

- Without preemption: long running goroutines can **starve** others
- Scheduler becomes **unfair**

**Go 1.14+**: sysmon gửi signal để preempt G chạy > 10ms

#### Hand off (Advanced)

Kỹ thuật nâng cao — khi M bị block (syscall), scheduler có thể "handoff" P sang M khác.

---

## gopark

`gopark` là runtime operation để **park (block) current goroutine**:

- Goroutine không runnable
- **Không consume CPU**
- Nằm trong waiting state

**Tự động gọi khi:**
- Channel receive → không có data
- Channel send → buffer full, không có receiver
- Mutex.Lock() → lock không available
- `runtime.Gosched()` → chủ động yield

---

## Context Switch Cost

### Cheap, nhưng có limit

- Mỗi goroutine có **stack** (dù nhỏ)
- **Stack scanning** during GC → overhead
- Goroutine không hoàn toàn miễn phí

### So sánh

| | Goroutine | OS Thread |
|---|---|---|
| Switch location | User-space | Kernel |
| Syscall | ❌ Không | ✅ Có |
| Stack size | ~2KB | 1–8MB |
| Speed | ~ns | ~μs |

---

## Goroutine Lifecycle

```
States of a Goroutine:

  [Created]
      │  go statement
      ↓
  _Runnable  ←─────────────────────────────────────────┐
      │  scheduled by P                                │
      ↓                                               unblock
  _Running ──────────── block (channel/mutex/syscall) → _Waiting
      │
  function returns
      ↓
   _Dead
```

### Goroutine States

| State | Ý nghĩa |
|-------|---------|
| `_Runnable` | Sẵn sàng chạy, đang chờ trong run queue của P |
| `_Running` | Đang thực thi trên một OS thread (M) |
| `_Waiting` | Blocked — channel, mutex, sleep, syscall, GC |
| `_Dead` | Đã kết thúc, stack có thể tái dụng |

## Goroutine vs OS Thread — Detailed Comparison

| | Goroutine | OS Thread |
|---|---|---|
| Creation cost | ~0.3-3μs, ~2-8KB stack | ~10-100μs, 1-8MB fixed stack |
| Context switch | ~100-200ns (user-space) | ~1-10μs (kernel trap) |
| Scheduler | Go runtime (user-space) | OS kernel |
| Stack type | Dynamic (grows/shrinks) | Fixed (stack overflow crash) |
| Max practical | 100K–1M | 1K–10K |
| Communication | Channel (CSP model) | Shared memory + lock |
| Preemption | ✅ Go 1.14+ (signal-based) | ✅ OS manages |

```go
// Benchmark: goroutine creation cost
func BenchmarkGoroutine(b *testing.B) {
    var wg sync.WaitGroup
    for i := 0; i < b.N; i++ {
        wg.Add(1)
        go func() { wg.Done() }()
    }
    wg.Wait()
}
// ~500ns-3μs per goroutine (vs ~50μs for pthread_create)
```

## GOMAXPROCS — Tuning Guide

```go
import "runtime"

procs := runtime.GOMAXPROCS(0)   // 0 = read current
fmt.Println("GOMAXPROCS:", procs) // default = num CPU cores

// Environment variable
// GOMAXPROCS=4 go run main.go
```

### Trade-off: GOMAXPROCS Settings

| Workload Type | Recommended | Lý do |
|--------------|------------|-------|
| CPU-bound | `= num CPUs` | Tận dụng tối đa parallelism |
| I/O-bound | `> num CPUs` | CPUs idle khi wait I/O → more goroutines useful |
| Low-latency | `1` | Eliminate context switch overhead |
| Containerized | Match cgroup CPU quota | Tránh over-subscription |

```bash
# ⚠️ Problem in Kubernetes:
# GOMAXPROCS thấy NODE CPUs (e.g., 64), không phải container limit (e.g., 2)
# → 64 OS threads cho 2 CPU cores → thrashing!

# ✅ Solution: automaxprocs
go get go.uber.org/automaxprocs

import _ "go.uber.org/automaxprocs"  // set GOMAXPROCS = cgroup limit
```

## Network Poller (netpoller)

Go tích hợp **async I/O network poller** trong runtime — OS syscall không block OS thread:

```
Network I/O Flow:
  goroutine
    │ conn.Read()
    │ → fd not ready
    │ → register fd with netpoller (epoll/kqueue)
    │ → gopark (goroutine blocked, M freed)
    ↓
  [other goroutines run on M]
    ↑
  OS event: fd ready
    │ → sysmon/pollWork unparks goroutine
    │ → goroutine back to run queue
    ↓
  goroutine.Read() returns data
```

→ **Một OS thread (M) có thể handle hàng nghìn concurrent connections** — foundation của Go's C10K capability.

## sysmon — Background Monitor

`sysmon` là goroutine đặc biệt của runtime (không cần P để chạy):

```
sysmon runs every 10–160ms (adaptive):
│
├── Preemption check
│   └── Goroutine chạy > 10ms → gửi SIGURG signal → force preempt
│
├── Retake P
│   └── M bị block trong syscall → detach P → P available cho khác
│
├── Network polling
│   └── Poll netpoller → wake goroutines với ready I/O
│
└── GC trigger
    └── GC chưa chạy đủ lâu → force GC cycle
```

## Concurrency vs Parallelism

> **"Concurrency is about dealing with many things at once. Parallelism is about doing many things at once."** — Rob Pike

| | Concurrency | Parallelism |
|---|---|---|
| Định nghĩa | Cấu trúc code để handle nhiều tasks | Thực sự execute nhiều tasks cùng lúc |
| Requires | Multiple goroutines | GOMAXPROCS > 1 + multi-core CPU |
| Single core | ✅ Có thể (interleaved) | ❌ Không thực sự |
| Example | Web server concurrent requests | Parallel matrix multiplication |

```go
// Concurrent (1 or more cores):
go task1()  // interleaved, take turns
go task2()

// Parallel (multi-core + GOMAXPROCS > 1):
// task1() và task2() run simultaneously on 2 different CPUs
```

## Work Stealing — Deep Dive

```
P0 run queue: [G1, G2, G3, G4]  ← nhiều goroutines
P1 run queue: []                 ← idle

Work Stealing:
  P1 thấy run queue rỗng
    → steal HALF goroutines từ P0
    → P1 run queue: [G3, G4]
    → P0 run queue: [G1, G2]
  → Load balanced!
```

> Work stealing tự động balance load giữa processors — programmer không cần quan tâm đến distribution.
