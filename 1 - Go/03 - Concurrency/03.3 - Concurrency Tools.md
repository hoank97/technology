# 03.3 Concurrency Tools

## Concurrency Pitfalls

### 1. Data Race

**2+ goroutines cùng access biến, ít nhất 1 write**

```go
// ❌ Data race
var counter int
go func() { counter++ }()
go func() { counter++ }()
fmt.Println(counter) // unpredictable
```

**Detection**: `go run -race main.go`

**Fix**:

```go
var mu sync.Mutex
mu.Lock(); counter++; mu.Unlock()

// Hoặc Atomic
var counter atomic.Int64
counter.Add(1)
```

### 2. Deadlock

**Tất cả goroutines chờ nhau**

```go
// ❌ Deadlock
ch := make(chan int)
<-ch    // blocked forever
ch <- 1 // blocked forever
```

### 3. Goroutine Leak

**Goroutine không exit, chiếm memory vĩnh viễn**

```go
// ❌ Leak
func leak() {
    ch := make(chan int)
    go func() { <-ch }() // block forever
}
```

### 4. Closing Channel Twice

```go
// ❌ Panic
ch := make(chan int)
close(ch)
close(ch) // panic: close of closed channel
```

### 5. Range Over Channel

```go
// Range tự động close khi channel đóng
for v := range ch {
    fmt.Println(v)
}
```

## sync.Mutex vs sync.RWMutex

### Mutex (Mutual Exclusion)

```go
var mu sync.Mutex

mu.Lock()
counter++
mu.Unlock() // Hoặc defer mu.Unlock()
```

### RWMutex (Read-Write Lock)

| Lock | Ai dùng | Mô tả |
|------|---------|--------|
| **RLock** | Reader | Đọc đồng thời được |
| **Lock** | Writer | Ghi độc quyền |

```go
var mu sync.RWMutex
var data map[string]string

// Read — nhiều goroutine đọc cùng lúc
mu.RLock()
v := data["key"]
mu.RUnlock()

// Write — phải exclusive
mu.Lock()
data["key"] = "value"
mu.Unlock()
```

→ **Tốt hơn Mutex** khi read >> write

## sync.Once

Đảm bảo code chỉ chạy **1 lần duy nhất** (thread-safe):

```go
var once sync.Once
var instance *DB

func GetInstance() *DB {
    once.Do(func() {
        instance = &DB{}
    })
    return instance
}
```

→ Thay thế cho Singleton pattern

## Atomic vs Mutex

### Atomic

Atomic operations là memory barrier đảm bảo sequential consistency — xem [[02.2 - Memory Model#Atomic Operations as Memory Barrier]].

```go
var counter atomic.Int64

counter.Add(1)
v := counter.Load()
counter.Store(100)
counter.CompareAndSwap(100, 101)
```

→ **Nhanh hơn Mutex** cho single variable
→ Chỉ work với primitive types (int, int32, int64, uint32, ...)

### Khi nào dùng?

| | Atomic | Mutex |
|---|---|---|
| Single variable | ✅ | ✅ |
| Multiple variables | ❌ | ✅ |
| Complex operations | ❌ | ✅ |

## Context — Deep Dive

### Core purpose

Handle **cancellation signals and deadlines** across API boundaries and goroutines. Context cancellation flow liên quan đến [[03.2 - Channel & Select#Done Channel Pattern]].

Allow a parent to signal all spawned children to **stop processing**, preventing resource waste.

### Operations

| | |
|---|---|
| `Done()` | Channel closed khi cancel/timeout |
| `Err()` | Return `Canceled` hoặc `DeadlineExceeded` |
| `Value(key)` | Lấy request-scoped value |

### Patterns

```go
// Cancel
ctx, cancel := context.WithCancel(context.Background())
go worker(ctx)
// ... later
cancel()

// Timeout
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()
data, err := fetchData(ctx)

// Value
ctx := context.WithValue(context.Background(), "userID", "123")
userID := ctx.Value("userID")
```

### Context WithCancelCause (Go 1.20+)

```go
import "errors"

func fetchWithCancel() {
    ctx, cancel := context.WithCancelCause(context.Background())
    go func() {
        err := doSomething()
        if err != nil {
            cancel(errors.New("connection failed"))
        }
    }()

    <-ctx.Done()
    fmt.Println(context.Cause(ctx)) // "connection failed"
}
```

### Context Deadline

```go
ctx, cancel := context.WithDeadline(context.Background(), time.Date(2024, 1, 1, 0, 0, 0, 0, time.UTC))
defer cancel()

select {
case <-ctx.Done():
    fmt.Println(ctx.Err()) // "context deadline exceeded"
case <-time.After(2 * time.Hour):
    fmt.Println("Task done")
}
```

## sync.Pool

### Purpose

**Object pooling** — reuse objects để giảm allocation và GC pressure — xem thêm [[02.1 - Memory & Runtime#Reducing GC Pressure — sync.Pool]].

```go
var bufPool = sync.Pool{
    New: func() any {
        return &bytes.Buffer{}
    },
}

func process(data string) {
    buf := bufPool.Get().(*bytes.Buffer)
    defer func() {
        buf.Reset()
        bufPool.Put(buf)
    }()

    buf.WriteString(data)
    // ... use buf
}
```

### Rules

| | Detail |
|---|--------|
| `Get()` | Returns nil if no items — không guaranteed nhận item đã Put |
| `Put()` | Optional — nếu không Put, object bị GC |
| GC | Pool **cleared** khi GC chạy — không dùng làm persistent cache |
| **Không phải** connection pool | Dùng `database/sql` cho connections |

### Common mistake

> sync.Pool bị **cleared mỗi GC cycle** — xem [[02.1 - Memory & Runtime#Reducing GC Pressure — sync.Pool]].

```go
// ❌ Không dùng Pool làm connection pool
var dbPool = sync.Pool{New: func() any { return sql.Open(...) }}
// → Mỗi GC cycle sẽ mất tất cả connections!

// ✅ Dùng database/sql — designed cho connection pooling
db, _ := sql.Open("postgres", "...")
db.SetMaxOpenConns(10)
```

## sync.Cond

### Purpose

**Condition variable** — goroutine **waiting for a condition** to become true.

```go
var mu sync.Mutex
cond := sync.NewCond(&mu)
ready := false

// Waiter
go func() {
    mu.Lock()
    defer mu.Unlock()
    for !ready {
        cond.Wait() // releases lock, blocks; re-acquires on Signal/Broadcast
    }
    fmt.Println("Ready!")
}()

// Signaler
mu.Lock()
ready = true
mu.Unlock()
cond.Signal() // wake ONE waiter (random)
```

### Methods

| Method | Khi nào dùng |
|--------|--------------|
| `Wait()` | Called **inside lock** — tự động releases lock, blocks |
| `Signal()` | Wake **one** waiter (FIFO trong Go implementation) |
| `Broadcast()` | Wake **all** waiters |
| `signal()` vs `Broadcast()` | Signal khi chỉ 1 waiter; Broadcast khi N waiters |

### ⚠️ Always use in a loop

```go
// ❌ Race condition
for condition {
    cond.Wait()
}

// ✅ Correct
mu.Lock()
for !condition {
    cond.Wait()
}
mu.Unlock()
```

### Cond vs Channel

| | sync.Cond | Channel |
|---|---|---|
| Lock | Cần external mutex | Không cần |
| Multiple waiters on same condition | ✅ Efficient | ❌ Cần N channels |
| Timeout | ❌ Manual | ✅ `time.After` |
| Signaling semantics | `Signal`/`Broadcast` | Send to specific channel |

## sync.Map

### Purpose

**High-read-concurrency map** — optimized cho N readers, infrequent writers.

```go
var sm sync.Map

// Store
sm.Store("key", "value")

// Load
val, ok := sm.Load("key")
fmt.Println(val, ok) // "value", true

// LoadOrStore
v, loaded := sm.LoadOrStore("key", "default")
fmt.Println(v, loaded) // "value", true (không insert "default")

// Delete
sm.Delete("key")

// Range — must be called serially
sm.Range(func(key, value any) bool {
    fmt.Println(key, value)
    return true // return false to stop
})
```

### vs map + Mutex

| Scenario | sync.Map | map + Mutex |
|----------|----------|------------|
| Read-heavy (R >> W) | ✅ Fast | ⚠️ Slow (lock on every read) |
| Balanced R/W | ⚠️ Slower than map+Mutex | ✅ Fast |
| Many keys, sparse | ✅ Better | ⚠️ Worse |
| Known key count | ⚠️ | ✅ Better |

```go
// ✅ sync.Map performance characteristic
// ~10x faster than Mutex for 90/10 read/write ratio
// Similar/slower for 50/50 ratio
```

### Common pitfalls

```go
// ❌ No len()
n := len(sm) // Compile error — sync.Map không có Len()

// ✅ Manual count
n := 0
sm.Range(func(k, v any) bool {
    n++
    return true
})

// ❌ Không dùng cho small maps (2-3 keys) — mutex simpler
// ✅ Dùng khi N goroutines read concurrently, rarely write
```

## atomic.Value — Arbitrary Type Store

`atomic.Value` lưu trữ một giá trị **bất kỳ type nào** một cách thread-safe, không cần type parameter (không dùng generics).

```go
var v atomic.Value

// Store — chỉ được gọi 1 lần (hoặc sau khi đọc xong)
v.Store(config)

// Load — đọc, an toàn đồng thời
loaded := v.Load()

// Swap — atomic replace, trả về old value
old := v.Swap(newConfig)

// CompareAndSwap (CAS) — chỉ replace nếu current == old
swapped := v.CompareAndSwap(oldConfig, newConfig)
```

### Safety rules

```go
// ❌ WRONG: Store interface type sau đó đọc type khác
var v atomic.Value
v.Store("string")           // ✅ OK
v.Load().(string)           // ✅ OK

// ❌ WRONG: Race giữa Store và Load của type khác
v.Store("string")
v.Load().([]int)  // ❌ Type assertion panic

// ✅ Store phải cùng type với Load
var v atomic.Value
type Config struct{ Timeout int }

v.Store(Config{Timeout: 30})
cfg := v.Load().(Config)

// ✅ Immutable updates: Store pointer/object không đổi
type Config struct {
    mu  sync.RWMutex
    vals map[string]any
}
v.Store(&Config{vals: map[string]any{}})

func update(k, val any) {
    cfg := v.Load().(*Config)
    cfg.mu.Lock()
    cfg.vals[k] = val
    cfg.mu.Unlock()
}
```

### atomic.Pointer[T] — generic atomic pointer (Go 1.19+)

```go
var ptr atomic.Pointer[Config]

// Store pointer
cfg := &Config{Timeout: 30}
ptr.Store(cfg)

// Load pointer
loaded := ptr.Load()

// Swap
old := ptr.Swap(newCfg)

// CAS
ok := ptr.CompareAndSwap(oldCfg, newCfg)
```

### atomic.Value vs atomic.Pointer[T] vs atomic.Int64

| Type | Best for | Not for |
|------|---------|--------|
| `atomic.Int64` / `atomic.Bool` | Counters, flags | Complex data |
| `atomic.Pointer[T]` | Single pointer | Multiple fields |
| `atomic.Value` | Config objects, arbitrary snapshots | Primitive types (dùng typed atomics) |

## sync.Mutex — Internals

### State encoding

`sync.Mutex` dùng **single integer** (`state` field) để encode cả lock status và waiter count:

| Bit | Meaning |
|-----|---------|
| `0` | Unlocked |
| `1` | Locked (fast path: atomic CAS) |
| `2+` | Waiter count |
| `mutexStarving` | Starvation mode bit (Go 1.9+) |

```go
// Pseudocode — mutex internal state
const (
    mutexLocked = 1 << iota // 1
    mutexWoken              // 2
    mutexStarving           // 4
    mutexWaitersShift = iota // 3
)
```

### Acquisition (fast path)

```go
// Fast path: atomic CAS
if atomic.CompareAndSwapInt32(&m.state, 0, 1) {
    return // acquired — no syscall
}
```

### Slow path (contended)

```go
// Slow path: sleep/wake
for {
    atomic.AddInt32(&m.state, 1)        // thêm waiter
    atomic.StoreInt32(&m.state, 1)     // try lock
    semacreate()                         // sleep on semaphore
}
```

### Starvation Mode (Go 1.9+)

Khi một waiter đợi > **1ms**, mutex chuyển sang **starvation mode**:
- FIFO cho tất cả waiters (fair — writer không bị starved)
- New lock requests **đợi ở queue cuối** (không nhảy vào fast path)

```go
// Starvation mode: fair to waiting writers
// Normal mode: better throughput, unfair to writers
```

### Profile mutex contention

```bash
go test -mutexprofile=mutex.out ./...
go tool pprof -http=:9090 mutex.out
```

## sync.WaitGroup — Deep Dive

`WaitGroup` chờ cho một nhóm goroutines hoàn thành. Không propagate error.

```go
var wg sync.WaitGroup

wg.Add(1)           // Báo có 1 goroutine sẽ chạy
go func() {
    defer wg.Done() // Giảm counter, khi counter = 0 → Wait unblocks
    doWork()
}()

wg.Add(1)
go func() {
    defer wg.Done()
    doMoreWork()
}()

wg.Wait()            // Block cho đến khi tất cả Done()
fmt.Println("all done")
```

### Add vs Done patterns

```go
// ❌ WRONG: Add sau khi goroutine đã start
go func() {
    wg.Add(1) // ❌ RACE CONDITION: Add() racing với Wait()
    defer wg.Done()
    doWork()
}()

// ✅ CORRECT: Add() phải trước goroutine
wg.Add(1)
go func() {
    defer wg.Done()
    doWork()
}()

// ✅ CORRECT: Add trong loop, trước goroutine
for _, task := range tasks {
    wg.Add(1)
    go func(t Task) {
        defer wg.Done()
        process(t)
    }(task)
}
wg.Wait()

// ✅ CORRECT: Add -1 pattern (tương đương Done)
for range tasks {
    wg.Add(-1) // Add nhận negative → decrement
    go func() {
        defer wg.Done()
        doWork()
    }()
}
```

### Add + Done in goroutine (conditional)

```go
// ⚠️ Chỉ dùng Add() trong goroutine KHI goroutine được bảo vệ bởi mutex
// hoặc khi Add() không racing với Wait()

// Pattern: Add trong goroutine chỉ OK khi Add() gọi trước Wait()
wg.Add(1)
go func() {
    // doWork() có thể panic → dùng recover + Done
    defer func() {
        if r := recover(); r != nil {
            log.Printf("worker panic: %v", r)
        }
        wg.Done()
    }()
    doWork()
}()
wg.Wait()
```

### WaitGroup vs Channel vs errgroup

| Aspect | `sync.WaitGroup` | Channel | `errgroup` |
|--------|-------------------|---------|------------|
| Error propagation | ❌ | ⚠️ | ✅ |
| Cancellation | ❌ | ⚠️ | ✅ (context) |
| goroutine management | ✅ own goroutines | ⚠️ | ✅ own goroutines |
| Use when | Fire-and-forget, no errors | Ownership transfer | Need errors/cancellation |

```go
// WaitGroup: khi không cần error
var wg sync.WaitGroup
for _, url := range urls {
    wg.Add(1)
    go func(u string) {
        defer wg.Done()
        http.Get(u) // ignore error
    }(url)
}
wg.Wait()

// errgroup: khi cần capture errors
g, ctx := errgroup.WithContext(ctx)
for _, url := range urls {
    url := url
    g.Go(func() error {
        resp, err := http.Get(url) // ❌ Context cancellation propagates up
        if err != nil {
            return err
        }
        return resp.Body.Close()
    })
}
if err := g.Wait(); err != nil {
    // at least one error
}
```

## Goroutine Leak Detection

### Runtime inspection

```go
import "runtime"

func TestNoLeak(t *testing.T) {
    start := runtime.NumGoroutine()

    // ... run code that should not leak ...

    end := runtime.NumGoroutine()
    if end > start {
        t.Errorf("goroutine leak: started %d, ended %d (+%d)",
            start, end, end-start)
    }
}
```

### Patterns that leak

Goroutine leak prevention — xem [[03.1 - Goroutine & Scheduler#Goroutine Lifecycle]].

```go
// ❌ Leaking: goroutine blocked on channel (never exits)
func leak1() {
    ch := make(chan int)
    go func() { <-ch }() // blocked forever — never exits
}

// ❌ Leaking: goroutine spinning forever
func leak2() {
    var wg sync.WaitGroup
    for {
        wg.Add(1)
        go func() {
            defer wg.Done()
            time.Sleep(time.Hour) // never stops
        }()
    }
}

// ✅ Correct: always have exit path
func noLeak() {
    ctx, cancel := context.WithCancel(context.Background())
    defer cancel()

    go worker(ctx)
}
```

### Debugging with pprof

```bash
# Goroutine profile
curl http://localhost:6060/debug/pprof/goroutine?debug=1

# Hoặc trong code
import _ "net/http/pprof"
go http.ListenAndServe(":6060", nil)
```

### go tool trace

```bash
go test -trace=trace.out ./...
go tool trace trace.out
# Browser visualization of goroutine lifecycle
```

### Prevention best practices

1. **Always pass `ctx`** — cancel khi done
2. **Always close channels** — signal completion
3. **Use `errgroup`** — collective cancellation
4. **Use WaitGroup** — wait for completion before returning

## Worker Pool Pattern

```go
// Classic worker pool: N workers, unlimited jobs via channel
func WorkerPool(ctx context.Context, numWorkers int, jobs <-chan Job) <-chan Result {
    results := make(chan Result, numWorkers)

    var wg sync.WaitGroup
    for i := 0; i < numWorkers; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            for {
                select {
                case <-ctx.Done():
                    return  // cancelled
                case job, ok := <-jobs:
                    if !ok {
                        return  // channel closed
                    }
                    results <- process(job)
                }
            }
        }()
    }

    go func() {
        wg.Wait()
        close(results)
    }()

    return results
}

// Usage
jobs := make(chan Job, 1000)
results := WorkerPool(ctx, runtime.NumCPU(), jobs)

for _, j := range allJobs {
    jobs <- j
}
close(jobs)

for r := range results {
    fmt.Println(r)
}
```

### Worker Pool Sizing

| Workload | Workers | Lý do |
|----------|---------|-------|
| CPU-bound | = `GOMAXPROCS` (= num CPUs) | Avoid context switch |
| I/O-bound | 2–10× `GOMAXPROCS` | CPUs idle during I/O |
| API rate-limited | = rate-limit slots | Don't exceed API quota |
| DB queries | Match DB connection pool | Bound by DB connections |

## Rate Limiting

```go
// Method 1: time.Ticker (token bucket)
func rateLimiter(ctx context.Context, rps int) func() bool {
    ticker := time.NewTicker(time.Second / time.Duration(rps))
    return func() bool {
        select {
        case <-ctx.Done():
            ticker.Stop()
            return false
        case <-ticker.C:
            return true
        }
    }
}

wait := rateLimiter(ctx, 10)  // 10 requests per second
for _, req := range requests {
    if !wait() {
        break
    }
    go process(req)
}

// Method 2: golang.org/x/time/rate (token bucket with burst)
import "golang.org/x/time/rate"

limiter := rate.NewLimiter(rate.Limit(10), 50)  // 10 rps, burst 50
if err := limiter.Wait(ctx); err != nil {
    return err  // ctx cancelled or deadline exceeded
}
// proceed
```

## Common Concurrency Anti-patterns

```go
// ❌ Anti-pattern 1: Goroutine leak (no exit path)
func handler() {
    ch := make(chan error)
    go func() {
        ch <- longOperation()  // if no one reads ch → goroutine stuck forever
    }()
    // forgot to read ch!!!
}
// ✅ Fix: buffered channel (goroutine won't block)
func handler() error {
    ch := make(chan error, 1)  // buffer 1 → goroutine won't block
    go func() { ch <- longOperation() }()
    return <-ch
}

// ❌ Anti-pattern 2: Mutex held across blocking channel op
mu.Lock()
ch <- value   // ❌ potential deadlock: receiver might also need mu
mu.Unlock()

// ✅ Fix: release lock before blocking operation
mu.Lock()
val := value
mu.Unlock()
ch <- val     // ✅ lock released before blocking

// ❌ Anti-pattern 3: Panic in goroutine kills entire program
go func() {
    panic("oops")  // ❌ kills whole program
}()

// ✅ Fix: recover in every goroutine
go func() {
    defer func() {
        if r := recover(); r != nil {
            log.Printf("goroutine panic: %v\n%s", r, debug.Stack())
        }
    }()
    doWork()
}()

// ❌ Anti-pattern 4: Shared mutable state without protection
var cache = map[string]Result{}
go func() { cache["a"] = compute() }()  // ❌ data race
go func() { _ = cache["a"] }()

// ✅ Fix: sync.Map or Mutex
var cache sync.Map
go func() { cache.Store("a", compute()) }()
go func() { cache.Load("a") }()
```

## 📌 Tóm tắt

```
Concurrency Tools
│
├── Pitfalls
│   ├── Data race → -race flag, Mutex/Atomic
│   ├── Deadlock → avoid circular lock order
│   ├── Goroutine leak → Context + buffered channel
│   └── Panic in goroutine → defer + recover
│
├── Mutex / RWMutex
│   ├── Mutex: exclusive (fast path CAS ~5ns, slow path lock)
│   ├── RWMutex: concurrent reads, exclusive writes
│   └── Starvation mode (Go 1.9+): FIFO fairness > 1ms
│
├── sync.Atomic
│   ├── ~5ns vs Mutex ~25ns
│   ├── Typed: atomic.Int64, atomic.Bool, atomic.Pointer[T]
│   └── ⚠️ Chỉ single values — ABA problem
│
├── Context
│   ├── WithCancel / WithTimeout / WithDeadline
│   ├── WithCancelCause (Go 1.20+): richer error
│   └── WithValue: request-scoped metadata
│
├── sync.Pool (reduce GC)
│   ├── Cleared each GC cycle
│   └── ⚠️ NOT for connections, sessions, long-lived objects
│
├── sync.Cond
│   ├── Wait in LOOP: for !condition { cond.Wait() }
│   └── Signal(1 waiter) vs Broadcast(all waiters)
│
├── sync.Map (concurrent map)
│   ├── Optimized for R >> W
│   └── No len(), Range is not atomic
│
├── Worker Pool Pattern
│   ├── N workers + job channel + results channel
│   ├── CPU-bound: N = GOMAXPROCS
│   └── I/O-bound: N = 2-10× GOMAXPROCS
│
├── Rate Limiting
│   ├── time.Ticker for simple RPS limiting
│   └── golang.org/x/time/rate for burst support
│
└── Sync Primitive Selection
    ├── Data transfer → Channel
    ├── Shared state → Mutex / RWMutex / Atomic
    ├── One-time → sync.Once
    ├── N goroutines → WaitGroup / errgroup
    └── Concurrent map → sync.Map
```
