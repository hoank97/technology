# 07.1 Testing & Benchmarking

## Overview

Go có `testing` package tích hợp sẵn — không cần library bên thứ 3 cho hầu hết use cases.

> Testing là first-class concern trong Go. Interview thường hỏi table tests, benchmarks, và mocking patterns.

---

## Testing.T — Basics

### Function signature

```go
// File: sum_test.go
func TestSum(t *testing.T) {
    // test logic
}
```

### Các methods cơ bản

| Method | Use case |
|--------|----------|
| `t.Fatal(...)` | Fail immediately — test không continue |
| `t.Fatalf("msg %d", val)` | Fail with formatted message |
| `t.Error(...)` | Fail but continue execution |
| `t.Errorf("msg %d", val)` | Fail with formatted, continue |
| `t.Skip("reason")` | Skip test (feature not implemented) |
| `t.SkipNow()` | Skip rest of test |

### Assertions pattern

```go
func AssertEqual[T any](t *testing.T, got, want T) {
    if got != want {
        t.Errorf("got %v, want %v", got, want)
    }
}

// Dùng
AssertEqual(t, Add(1, 2), 3)
```

## Table-Driven Tests

### Pattern chuẩn

```go
func TestAdd(t *testing.T) {
    tests := []struct {
        name string
        a, b int
        want int
    }{
        {"zero", 0, 0, 0},
        {"positive", 1, 2, 3},
        {"negative", -1, -1, -2},
        {"large", 1_000_000, 2_000_000, 3_000_000},
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            got := Add(tt.a, tt.b)
            if got != tt.want {
                t.Errorf("Add(%d, %d) = %d; want %d", tt.a, tt.b, got, tt.want)
            }
        })
    }
}
```

**Output mẫu**:
```
=== RUN   TestAdd/zero
--- PASS: TestAdd/zero (0.00s)
=== RUN   TestAdd/positive
--- PASS: TestAdd/positive (0.00s)
=== RUN   TestAdd/negative
--- PASS: TestAdd/negative (0.00s)
=== RUN   TestAdd/large
--- PASS: TestAdd/large (0.00s)
--- PASS: TestAdd (4 subtests)
```

### Tại sao table tests?

| | Individual tests | Table tests |
|---|---|---|
| Thêm test case | Thêm function mới | Thêm row vào slice |
| Shared setup | Copy-paste | Dùng chung setup trong loop |
| Consistency | Tùy developer | Uniform pattern |
| Scalability | ❌ | ✅ |

## Subtests với `t.Run()`

```go
func TestDatabase(t *testing.T) {
    // Setup chung
    db := setupDB(t) // helper with cleanup

    t.Run("insert", func(t *testing.T) {
        err := db.Insert("key", "value")
        AssertNoError(t, err)
    })

    t.Run("query", func(t *testing.T) {
        val, err := db.Query("key")
        AssertEqual(t, val, "value")
    })

    t.Run("delete", func(t *testing.T) {
        err := db.Delete("key")
        AssertNoError(t, err)
    })
}
```

### Cleanup sau subtest

```go
func TestFile(t *testing.T) {
    // per subtest cleanup
    t.Run("write", func(t *testing.T) {
        f := createTempFile(t, "test.txt")
        defer os.Remove(f.Name())
        // test
    })
}
```

## HTTP Testing với `httptest`

### Test HTTP Handler

```go
func TestHandler(t *testing.T) {
    // Tạo HTTP request giả
    req := httptest.NewRequest(http.MethodGet, "/hello", nil)
    // Tạo ResponseRecorder để capture response
    rec := httptest.NewRecorder()

    // Gọi handler
    Handler(rec, req)

    // Assertions
    AssertEqual(t, rec.Code, http.StatusOK)

}

## Testing Errors

```go
func TestErrors(t *testing.T) {
    t.Run("not found", func(t *testing.T) {
        err := db.Query("nonexistent")
        AssertEqual(t, err, ErrNotFound)
    })

    t.Run("invalid input", func(t *testing.T) {
        err := validate("")
        AssertError(t, err, "empty input")
    })
}

// Helper
func AssertError(t *testing.T, err error, substr string) {
    if err == nil {
        t.Fatal("expected error, got nil")
    }
    if !strings.Contains(err.Error(), substr) {
        t.Errorf("error %q does not contain %q", err.Error(), substr)
    }
}
```

## Benchmarking với `testing.B`

### Basic benchmark

```go
func BenchmarkFibonacci(b *testing.B) {
    for i := 0; i < b.N; i++ {
        Fibonacci(20)
    }
}
```

### Chạy benchmarks

```bash
go test -bench=. -benchmem ./...
go test -bench=BenchmarkFibonacci -benchmem ./...
go test -bench=BenchmarkFibonacci -bench=10 -count=5 ./...
```

### Output mẫu

```
BenchmarkFibonacci-8    100000000    12.3 ns/op    0 B/op    0 allocs/op
```

| Column | Ý nghĩa |
|--------|---------|
| `100000000` | Test chạy `b.N` lần — tự động scale |
| `12.3 ns/op` | Trung bình mỗi operation |
| `0 B/op` | Memory allocate mỗi operation |
| `0 allocs/op` | Số allocations mỗi operation |

### Báo cáo allocations

```go
func BenchmarkLargeAlloc(b *testing.B) {
    b.ReportAllocs()
    for i := 0; i < b.N; i++ {
        _ = make([]byte, 1024)
    }
}
```

### So sánh benchmarks

```bash
go test -bench=. -benchmem -cpuprofile=cpu.out ./...
go test -bench=. -benchmem -memprofile=mem.out ./...

# Visualize
go tool pprof -http=:9090 cpu.out
```

### Realistic iteration

```go
// ❌ Sai — benchmarks quá nhanh
func BadBenchmark(b *testing.B) {
    for i := 0; i < 1000; i++ {
        Fibonacci(20) // benchmark mất <1ms
    }
}

// ✅ Đúng — để framework quyết định số iterations
func GoodBenchmark(b *testing.B) {
    for i := 0; i < b.N; i++ {
        Fibonacci(20) // b.N tự động scale
    }
}

## Mocking Patterns

### Interface-based mocking

Build tags cho integration tests — xem thêm [[07.2 - Build & Tooling#Build Constraints]].

```go
// Interface defines contract
type UserStore interface {
    GetUser(id string) (*User, error)
    SaveUser(u *User) error
}

// Real implementation
type PostgresStore struct{ db *sql.DB }

func (s *PostgresStore) GetUser(id string) (*User, error) {
    // ...
}

// Mock implementation
type MockStore struct {
    users map[string]*User
}

func (m *MockStore) GetUser(id string) (*User, error) {
    return m.users[id], nil
}
func (m *MockStore) SaveUser(u *User) error {
    m.users[u.ID] = u
    return nil
}

// Trong test
func TestService(t *testing.T) {
    store := &MockStore{users: map[string]*User{
        "1": {ID: "1", Name: "Alice"},
    }}
    svc := NewService(store)
    // ...
}
```

## Testing Pyramid — Góc Nhìn Đa Chiều

```
E2E Tests (few)
      △     Slow, expensive, fragile. Test whole system.
     △△△    Run in CI only. Catch integration regressions.
    △△△△△
   ───────
Integration Tests (moderate)
  Test interaction giữa các components (DB, HTTP, etc.)
  Cần setup/teardown. Slower than unit. More realistic.
   ───────────────────
Unit Tests (many)
Fast, isolated, deterministic. Test logic thuần túy.
Run on every file save. Highest ROI for business logic.
```

### Trade-off: Bao nhiêu mỗi loại?

| | Unit | Integration | E2E |
|---|---|---|---|
| Speed | ✅ Fast (<10ms) | ⚠️ Slow (100ms-5s) | ❌ Very slow (5s-min) |
| Isolation | ✅ High | ⚠️ Medium | ❌ Low |
| Maintenance | ✅ Easy | ⚠️ Medium | ❌ Hard |
| Confidence | ⚠️ Low (mocked) | ✅ High | ✅✅ Highest |
| Recommended % | 70% | 20% | 10% |

## Fuzz Testing (Go 1.18+)

```go
// Fuzzing: compiler tự generate inputs để tìm crash/panic
func FuzzParseJSON(f *testing.F) {
    // Seed corpus — known interesting inputs
    f.Add(`{"name":"alice","age":30}`)
    f.Add(`{}`)
    f.Add(`null`)

    f.Fuzz(func(t *testing.T, data string) {
        // Function phải không panic với ANY input
        // Compiler sẽ generate random + mutated inputs
        _, _ = parseJSON(data)
        // Nếu panic → fuzz finds the bug!
    })
}

// Run fuzzing:
// go test -fuzz=FuzzParseJSON -fuzztime=30s
// Tìm thấy bug → lưu vào testdata/fuzz/FuzzParseJSON/...
// Next run: testdata automatically used as corpus
```

### ✅ Khi nào nên fuzz?

- Parser (JSON, XML, binary protocol)
- Input validation functions
- Cryptographic operations
- Any function processing untrusted input

## Test Isolation Patterns

```go
// Pattern 1: t.TempDir() — auto cleanup temp files
func TestWriteFile(t *testing.T) {
    dir := t.TempDir() // auto removed after test
    path := filepath.Join(dir, "test.txt")
    err := writeFile(path, "hello")
    assert.NoError(t, err)
}

// Pattern 2: t.Setenv() — auto restore env vars
func TestConfig(t *testing.T) {
    t.Setenv("DB_HOST", "localhost")
    // Env restored after test, even on failure
    cfg := loadConfig()
    assert.Equal(t, "localhost", cfg.DBHost)
}

// Pattern 3: t.Cleanup() — arbitrary cleanup
func TestWithResource(t *testing.T) {
    resource := acquireResource()
    t.Cleanup(func() {
        resource.Release() // runs after test, even on failure
    })
    // test body
}

// Pattern 4: Dependency injection for testability
type Service struct {
    db  DB       // interface, not *sql.DB
    now func() time.Time  // injectable clock
}

func NewService(db DB) *Service {
    return &Service{db: db, now: time.Now}
}

// In test:
svc := &Service{
    db:  &mockDB{},
    now: func() time.Time { return fixedTime },
}
```

## stdlib testing vs testify — Trade-offs

```go
// stdlib only (verbose but zero dependency)
func TestAdd(t *testing.T) {
    result := add(2, 3)
    if result != 5 {
        t.Errorf("add(2, 3) = %d, want 5", result)
    }
}

// testify (concise, better error messages)
import "github.com/stretchr/testify/assert"
func TestAdd(t *testing.T) {
    assert.Equal(t, 5, add(2, 3))
    // → "Expected: 5, Got: 4" (clear diff)
}

// testify require (stop on first failure)
import "github.com/stretchr/testify/require"
func TestService(t *testing.T) {
    user, err := service.GetUser(1)
    require.NoError(t, err)   // stops if error (unlike assert)
    require.NotNil(t, user)
    assert.Equal(t, "Alice", user.Name)  // continues even if fail
}
```

| | stdlib | testify |
|---|---|---|
| Dependency | ✅ None | ⚠️ External |
| Error messages | ⚠️ Manual formatting | ✅ Auto diff |
| Assertion variety | ⚠️ Limited | ✅ Rich |
| Mock support | ❌ | ✅ testify/mock |
| Community | ✅ Universal | ✅ Very popular |
| Recommended | Simple projects | Most projects |

## Test Coverage — Strategic Thinking

```bash
# Generate coverage report
go test -coverprofile=coverage.out ./...
go tool cover -html=coverage.out  # visual HTML

# Coverage by function
go tool cover -func=coverage.out

# CI threshold
go test -coverprofile=coverage.out ./...
coverage=$(go tool cover -func=coverage.out | grep total | awk '{print $3}' | tr -d '%')
if [ "$coverage" -lt "80" ]; then exit 1; fi
```

### Coverage Trade-offs

| Coverage % | Interpretation |
|-----------|---------------|
| 30-50% | Basic smoke tests only |
| 60-70% | Decent, happy-path covered |
| 80-90% | Good — error paths covered |
| 95-100% | Diminishing returns — may include boilerplate |

> **Anti-pattern**: Optimize for coverage % → write useless tests just to hit metric. Focus on testing **critical paths** and **edge cases**, not 100% line coverage.

// Benchmark with different inputs (sub-benchmarks)
func BenchmarkSort(b *testing.B) {
    sizes := []int{10, 100, 1000, 10000}
    for _, size := range sizes {
        b.Run(fmt.Sprintf("size=%d", size), func(b *testing.B) {
            data := make([]int, size)

## benchstat — Statistical Benchmark Comparison

`benchstat` (golang.org/x/tools/cmd/benchstat) so sánh kết quả benchmark một cách **thống kê** — xác định xem thay đổi có thực sự cải thiện hay chỉ là noise.

```bash
go install golang.org/x/tools/cmd/benchstat@latest

# Chạy benchmark nhiều lần để lấy mẫu
go test -bench=. -benchmem -count=10 > before.txt
# ... thay đổi code ...
go test -bench=. -benchmem -count=10 > after.txt

# So sánh
benchstat before.txt after.txt
```

### Interpreting output

```text
name     old ns/op     new ns/op     delta
Fib-10   200.5±0.5    198.2±0.3     -1.15%  (p=0.000)
              ↑             ↑
        baseline      after change
                    ±0.3 = standard deviation
                    p=0.000 = statistically significant (p < 0.05)
```

| p-value | Meaning |
|---------|---------|
| `p < 0.05` | Statistically significant — có thật sự khác biệt |
| `p > 0.05` | Không có ý nghĩa thống kê — có thể chỉ là noise |

### Statistical significance

```bash
# Single benchmark comparison
go test -bench=Fib -benchmem -count=30 > fib.txt
benchstat fib.txt

# Key output:
# ns/op: nanoseconds per operation
# B/op: bytes allocated per operation
# allocs/op: number of allocations per operation
# ±: standard deviation (SD)
# p: p-value (probability difference is random)
```

### Common patterns

```bash
# Baseline comparison (before vs after)
go test -bench=. -benchmem -count=20 > /tmp/before.txt
# modify code
go test -bench=. -benchmem -count=20 > /tmp/after.txt
benchstat /tmp/before.txt /tmp/after.txt

# Multiple systems comparison (e.g., different Go versions)
go test -bench=. -benchmem -count=20 > go1.21.txt
GOTOOLCHAIN=local go test -bench=. -benchmem -count=20 > go1.22.txt
benchstat go1.21.txt go1.22.txt

# CI automation — fail nếu regression > 5%
benchstat /tmp/baseline.txt /tmp/after.txt | grep -q regression
if [ $? -eq 0 ]; then echo "REGRESSION DETECTED"; exit 1; fi
```

### Tips

```bash
# -count càng cao → kết quả càng ổn định
# 5: quick check
# 10: standard for CI
# 30: for publication-quality results

# -benchtime càng cao → mỗi benchmark chạy lâu hơn
go test -bench=. -benchtime=10s -count=5
# Mỗi benchmark chạy tối thiểu 10 giây
```
