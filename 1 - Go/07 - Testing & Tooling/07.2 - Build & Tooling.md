# 07.2 Build & Tooling

## Overview

Các công cụ và flags quan trọng để build, debug, và optimize Go applications.

## Build Constraints

### `//go:build` (Go 1.17+) vs `// +build` (legacy)

```go
// ✅ Go 1.17+: Boolean expressions
//go:build linux && amd64
package linux64

// ✅ Multiple conditions
//go:build (darwin || linux) && !arm
package nonArm

// ❌ Legacy syntax (Go < 1.17)
// +build linux
// +build amd64
```

### Common constraints

| Constraint | Meaning |
|------------|---------|
| `//go:build linux` | Linux only |
| `//go:build !windows` | Not Windows |
| `//go:build go1.21` | Go 1.21+ only |
| `//go:build ignore` | File never compiled (for code gen) |

### Build tags cho tests

```go
//go:build integration
package integration_test

// Chạy: go test -tags=integration ./...
```

## Compiler Directives

### `//go:noinline`

```go
// ❌ Prevent function inlining — dùng cho precise benchmarking
func ImportantFunction() int {
    //go:noinline
    result := computeSomething()
    return result
}
```

### `//go:nosplit`

```go
// ❌ Skip stack split check — DANGEROUS, kernel-level code only
// Avoids check before calling another function
func NoSplitFunc() {
    //go:nosplit
    // ONLY use with extreme caution
}
```

### `//go:norace`

```go
// Disable race detector for specific function
func NoRaceCheck() {
    //go:norace
    // Race detector won't instrument this function
}
```

## Runtime Functions

### `runtime.LockOSThread()`

**Pin goroutine to specific OS thread** — goroutine không bị schedule sang thread khác.

```go
func init() {
    runtime.LockOSThread()
    // This goroutine is now permanently bound to a thread
    // Use for: cgo callbacks, GPU operations, FPU-intensive code
}
```

**Khi nào dùng?**

| Use case | Tại sao |
|----------|---------|
| cgo callbacks | OS thread context cần preserved |
| Real-time / gaming | Deterministic scheduling |
| GPU drivers | Thread affinity requirements |
| `exec.Cmd` + StdinPipe | stdin cần same thread |

```go
func worker() {
    runtime.LockOSThread()
    defer runtime.UnlockOSThread()

    // This goroutine runs on one OS thread
    // OS thread won't be used by other goroutines
    doWork()
}
```

### `runtime.Gosched()`

**Voluntarily yield** execution slot — cho phép scheduler chạy other runnable goroutines.

```go
func example() {
    go func() {
        for i := 0; i < 1000; i++ {
            doWork(i)
        }
    }()

    runtime.Gosched() // yield → let other goroutines run
    // ...
}
```

**Use cases**: cooperative multitasking, preventing starvation in long loops.

### `runtime.GOMAXPROCS()`

Set số OS threads active for scheduling:

```go
// Giới hạn concurrency
runtime.GOMAXPROCS(1) // Sequential (1 goroutine at a time)
// Common: benchmark isolated single-thread performance

runtime.GOMAXPROCS(0) // Reset to default (CPU cores)
```

### `runtime.NumGoroutine()`

```go
import "runtime"

n := runtime.NumGoroutine()
fmt.Println("Active goroutines:", n)
```

**Use case**: leak detection trong tests.

## Build Flags

### Reproducible builds

```bash
go build -trimpath ./...
# Remove file system paths from binary
# Binary identical across machines
```

### Strip debug symbols

```bash
go build -ldflags="-s -w" ./...
# -s: strip symbol table
# -w: strip DWARF info
# → ~30% smaller binary
```

### Version / Metadata injection

```bash
go build -ldflags="
  -X main.Version=1.2.3
  -X main.GitCommit=$(git rev-parse HEAD)
  -X main.BuildTime=$(date -u)
" ./cmd/app
```

```go
var Version string
var GitCommit string
var BuildTime string

func main() {
    fmt.Printf("Version: %s, Commit: %s, Built: %s\n",
        Version, GitCommit, BuildTime)
}
```

### Cross-compilation

```bash
# Linux on Mac
GOOS=linux GOARCH=amd64 go build -o app-linux ./cmd/app

# Windows on Mac
GOOS=windows GOARCH=amd64 go build -o app.exe ./cmd/app

# ARM (Raspberry Pi)
GOOS=linux GOARCH=arm GOARM=7 go build -o app-pi ./cmd/app
```

## Performance Profiling

### pprof

pprof cho memory/CPU profiling — liên quan đến [[02.1 - Memory & Runtime#Memory Allocator (Malloc)]].

```bash
# CPU profile
go test -cpuprofile=cpu.out -bench=. ./...
go tool pprof cpu.out
(pprof) top
(pprof) web  # open browser graph

# Memory profile
go test -memprofile=mem.out -bench=. ./...
go tool pprof mem.out

# Goroutine profile
go test -mutexprofile=mutex.out -race ./...
go tool pprof mutex.out
```

### HTTP pprof endpoint

```go
import _ "net/http/pprof"

// Trong main()
go http.ListenAndServe(":6060", nil)
```

```bash
curl http://localhost:6060/debug/pprof/  # profiles list
curl http://localhost:6060/debug/pprof/goroutine?debug=1  # text dump
curl http://localhost:6060/debug/pprof/heap  # heap profile
```

### trace tool

```bash
go test -trace=trace.out ./...
go tool trace trace.out
# Browser: goroutine lifecycle, scheduling, syscalls, GC
```

## Benchmarking & Optimization Loop

GOMAXPROCS tuning — liên quan đến [[03.1 - Goroutine & Scheduler#GOMAXPROCS — Tuning Guide]].

```
1. Write benchmark (testing.B)
2. go test -bench=. -benchmem
3. Identify bottleneck (hot function)
4. Optimize or add //go:noinline if needed
5. Re-benchmark
6. Profile with -cpuprofile if needed
7. go tool pprof cpu.prof
8. Repeat
```

## Profiling Deep-Dive — pprof

```go
import (
    "net/http"
    _ "net/http/pprof"  // registers /debug/pprof/* endpoints
    "runtime/pprof"
)

// Method 1: HTTP endpoint (for long-running services)
func main() {
    go func() {
        http.ListenAndServe(":6060", nil)  // pprof server
    }()
    // ...
}
// curl http://localhost:6060/debug/pprof/ → list profiles
// go tool pprof http://localhost:6060/debug/pprof/heap

// Method 2: File-based profiling
func profileCPU() {
    f, _ := os.Create("cpu.prof")
    pprof.StartCPUProfile(f)
    defer pprof.StopCPUProfile()
    // ... code to profile
}

// Method 3: Test profiling
// go test -cpuprofile=cpu.prof -memprofile=mem.prof -bench=.
```

### pprof Profile Types

| Profile | Flag | Shows |
|---------|------|-------|
| CPU | `-cpuprofile` | Where CPU spends time |
| Heap | `-memprofile` | Live + historical allocations |
| Goroutine | via HTTP | All goroutine stacks |
| Block | `-blockprofile` | Where goroutines block on sync |
| Mutex | `-mutexprofile` | Mutex contention hotspots |
| Trace | `-trace` | Timeline: GC, scheduler, goroutines |

```bash
# Interactive pprof analysis
go tool pprof -http=:9090 cpu.prof
# → Opens browser with flame graph, call graph, treemap

# Key commands in CLI mode:
(pprof) top10    # top 10 functions by self time
(pprof) list FuncName  # annotated source code
(pprof) web      # open flame graph
```

## Profile-Guided Optimization (PGO) — Go 1.21+

```bash
# Step 1: Build normally
go build -o myapp .

# Step 2: Run with CPU profile
./myapp &
curl http://localhost:6060/debug/pprof/profile?seconds=30 > default.pgo

# Step 3: Build with PGO — compiler uses profile to optimize hot paths
go build -pgo=default.pgo -o myapp_optimized .
# → ~3-7% performance improvement for typical workloads
```

> **PGO internals**: compiler inlines functions called frequently during profiling, deoptimizes cold paths → net speedup without code changes.

## Build Size Optimization

```bash
# Check binary size
go build -o app . && ls -lh app

# Strip debug symbols — smaller binary
go build -ldflags="-s -w" -o app .
# -s: strip symbol table
# -w: strip DWARF debug info
# Typical reduction: 30-40%

# UPX compression (post-build)
upx --best app  # ~60-70% reduction, slower startup

# Size analysis
go tool nm app | sort -k2 -rn | head  # largest symbols
govulncheck ./...  # check for vulnerabilities
gosize ./...       # size analysis per package
```

### Binary Size Trade-offs

| Technique | Size Reduction | Downsides |
|-----------|---------------|----------|
| `-ldflags="-s -w"` | 30-40% | No stack traces in panics |
| UPX compression | 60-70% | Slow decompression on startup |
| Dead code elim | 5-15% | Auto (linker does it) |
| No CGo | 10-20% | Can't use C libraries |

## cgo — Trade-offs

```go
// cgo cho phép gọi C code từ Go
/*
#include <stdio.h>

void printHello() {
    printf("Hello from C!\n");
}
*/
import "C"

func main() {
    C.printHello()
}
```

### cgo Trade-offs — Thường là Bad Idea

| | cgo ✅ | cgo ❌ |
|---|---|---|
| Access C libraries | ✅ Only way | — |
| Cross-compilation | — | ❌ Breaks `CGO_ENABLED=0` |
| Build time | — | ❌ Much slower |
| Binary portability | — | ❌ Libc dependency |
| Race detector | — | ❌ Partial |
| goroutine switch | — | ❌ Can't switch M during cgo call |
| Prefer | When C lib unavailable in Go | Most cases — use pure Go |

```bash
# Pure Go build (no cgo) — more portable
CGO_ENABLED=0 go build .
# Uses Go's internal DNS resolver, TLS, etc.
# Works in minimal Docker images (FROM scratch)
```

## Build vs Compiler Directive Decision

```go
// //go:build — controls WHICH files compile
//go:build linux && amd64

// vs //go:generate — RUNS commands during build
//go:generate protoc --go_out=. proto/service.proto

// vs //go:embed — EMBEDS files into binary
//go:embed static/*
var staticFiles embed.FS

// vs //go:noinline — PREVENTS compiler optimization
//go:noinline
func criticalFunc() { ... }  // force no inlining (for benchmarking)

// vs //go:linkname — ACCESS unexported symbols
//go:linkname runtime_schedule runtime.schedule
func runtime_schedule()  // dangerous, unstable
```

## Production Build Checklist

```bash
# 1. Run all checks
go vet ./...
staticcheck ./...
go test -race -cover ./...
govulncheck ./...

# 2. Build with metadata injection
go build \
  -ldflags="-s -w -X main.version=$(git describe --tags) -X main.buildTime=$(date -u +%Y%m%dT%H%M%SZ)" \
  -trimpath \          # reproducible build (no local paths)
  -o app .

# 3. Docker build (multi-stage, minimal image)
# FROM golang:1.22 AS builder
# COPY . .
# RUN CGO_ENABLED=0 go build -o /app .
# FROM scratch          ← minimal image
# COPY --from=builder /app /app
# ENTRYPOINT ["/app"]
```
