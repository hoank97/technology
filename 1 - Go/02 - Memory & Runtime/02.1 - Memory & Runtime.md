# 02.1 Memory & Runtime



- Starts **small (~2KB)**
- **Grows and shrinks automatically**
- No fixed size như C

| | C | Go |
|---|---|---|

## Escape Analysis

Escape analysis quyết định biến nằm **stack** hay **heap**.

### Goals

- Dùng **stack** khi có thể (nhanh, không GC)
- Dùng **heap** chỉ khi cần (flexible, GC managed)

### Khi nào biến "escape" (sang Heap)?

1. **Return pointer của biến local**
   ```go
   func f() *int {
       n := 10       // n escapes vì pointer trả ra ngoài
       return &n
   }
   ```

2. **Share giữa goroutines**
   ```go
   go func() { fmt.Println(*s) }() // s escapes
   ```

3. **Interface boxing**
   ```go
   var i interface{} = &x  // x escapes vì boxing
   ```

### Kiểm tra

```bash
go run -gcflags '-m' main.go
```

## Garbage Collector

### How it works

- Automatically frees **heap memory** that is no longer reachable
- **Concurrent Mark & Sweep**
  - GC finding unused objects
  - Reclaiming memory safely

### GC Tuning

```go
// GOGC (default: 100)
 // Trigger khi heap tăng thêm 100%
runtime.GOGC = 50   // Aggressive: ít memory hơn
runtime.GOGC = 200  // Conservative: ít GC hơn
```

### Stop the World

- Go GC is **mostly concurrent**
- Vẫn cần **brief pauses** (mark termination)
- ~100–500μs pause — thường không đáng kể

## GC Deep Dive — Tri-Color Algorithm

### Bối cảnh

Go GC sử dụng **tri-color mark-and-sweep** algorithm — cho phép marking đồng thời (concurrent) mà không dừng chương trình.

### Ba màu (trạng thái)

| Trạng thái | Màu | Ý nghĩa |
|------------|-----|---------|
| **White (trắng)** | ❌ | "Chưa được check" — objects tiềm năng là garbage |
| **Grey (xám)** | ⚪ | "Đã thấy nhưng chưa explore children" — work queue |
| **Black (đen)** | 🟢 | "Đã check hoàn toàn" — alive, không bị thu hồi |

### Các bước (GC Cycle)

```
1. SWEEP: Clear all white objects (reclaim memory)
2. MARK: Đổi tất cả white → white (bắt đầu)
3. MARK: GC root traversal
   - Bắt đầu từ "roots" (stack, global vars, registers)
   - White(root) → Grey(root) (đưa vào queue)
   - Loop:
     • Pop grey object
     • Mark all its children white→grey (add to queue)
     • Grey → Black (done)
   - Khi queue rỗng → tất cả reachable objects là black
4. SWEEP: Tất cả white objects = garbage → reclaim
```

### Tại sao cần 3 màu?

> Nếu chỉ dùng 2 màu (checked/unchecked), không thể làm concurrent — GC có thể "miss" objects được allocated trong lúc marking.

### Write Barrier (Rào cản ghi)

**Vấn đề**: Khi GC đang marking (concurrent), một goroutine có thể **write pointer** vào object màu trắng (chưa check) → object đó bị mất (garbage nhưng không bị thu hồi).

**Giải pháp**: **Dijkstra-style write barrier** — khi goroutine ghi pointer, GC được notify:

```go
// Pseudocode — xảy ra khi ghi pointer vào field
writeBarrier(ptr *Object, field *Object) {
    // Nếu object được ghi vào đang ở trạng thái white
    // → đẩy nó thành grey (để check sau)
    if ptr.color == WHITE {
        ptr.color = GREY
        pushToQueue(ptr)
    }
    *field = ptr
}
```

→ Đảm bảo no objects bị mất khi GC chạy concurrent.

### GC Phases tổng quan

```
┌─────────────────────────────────────────────┐
│  OFF (STW)  ← Program running normally     │
│    ↓                                        │
│  MARK (Concurrent) ← GC scan roots          │
│    ↓                                        │
│  MARK TERMINATION (STW, ~100-500μs)        │
│    ↓                                        │
│  SWEEP (Concurrent) ← Reclaim white objs   │
│    ↓                                        │
│  OFF  ← Trở về normal                       │
└─────────────────────────────────────────────┘
```

### GC Pacer — Khi nào trigger?

Go không trigger GC ở fixed interval. **GC Pacer** quyết định:

```
Trigger khi: heap_used ≥ heap_goal × (1 + 100/GOGC)

Ví dụ:
- GOGC=100 (default): trigger khi heap tăng 100% (×2)
- GOGC=50: trigger khi heap tăng 50% (×1.5) → chạy GC sớm hơn
- GOGC=200: trigger khi heap tăng 200% (×3) → chạy GC muộn hơn
```

### GC Tuning Tips

```go
// Observing GC behavior
// $ GODEBUG=gctrace=1 go run main.go
// Output:
// gc 3 @0.012s @0.018s: 0.001+0.001+0.001 ms, 0 -> 1 MB

// Format: gc # @wall_time @cpu_time: pause_ms mark_ms sweep_ms heapsize -> heapsize

// Disable GC hoàn toàn (debug only!)
// $ GOGC=off go run main.go
// ⚠️ Memory sẽ grow vô hạn!
```

### STW Pauses — Tại sao ngắn?

| Component | Thời gian |
|-----------|-----------|
| Mark termination (STW) | ~100–500μs |
| Write barrier overhead | Trên mỗi pointer write (~2% CPU) |

Go 1.5+ dùng **hybrid write barrier** giảm STW từ ~10ms → ~100-500μs.

### Memory Allocator (Malloc)

| Layer | Chi tiết |
|-------|----------|
| **mcache** | Per-P (processor) cache, lock-free |
| **mcentral** | Shared between P, size-class based |
| **mheap** | Global arena, span-based allocation |

```
malloc flow: mcache (no lock) → mcentral (lock) → mheap (lock)
```

## Escape Analysis Deep Dive

### Kiểm tra chi tiết

```bash
go run -gcflags '-m=2' main.go   # 2 levels = show reasons
```

### Chi tiết từng trường hợp

```go
// ✅ NO ESCAPE — không trả pointer ra ngoài
func sum(a, b int) int {
    return a + b
}

// ❌ ESCAPE — trả pointer của local variable
func getInt() *int {
    n := 10       // n allocated on STACK, but pointer escapes
    return &n     // → n MUST be on HEAP (can't return stack pointer)
    // Compiler: "moved to heap: n"
}

// ✅ NO ESCAPE — slice được pass by value nhưng elements được copy
func process(vals []int) {
    // vals được copy (pointer + len + cap)
    // nhưng backing array không escape vì không return
}

// ❌ ESCAPE — interface boxing
func printValue(v interface{}) {
    fmt.Println(v) // v is boxed into interface{}
}

type User struct{ Name string }
u := User{Name: "Alice"}
printValue(u)   // u escapes vì boxed into interface{}
printValue(&u)  // *u escapes vì boxed
printValue(42)  // 42 escapes vì boxed
```

### Compiler hints

```go
// ❌ Ngăn function được inlined (dùng cho precise benchmarking)
func NoInline() int {
    //go:noinline
    result := 42
    return result
}

// Ngăn function bị escape analysis
func NoEscape() *int {
    //go:noescape
    // Note: chỉ áp dụng cho một số function nhất định
}
```

### Khi nào Escape quan trọng?

| Scenario | Tại sao cần hiểu |
|----------|-------------------|
| Performance | Heap allocation chậm hơn stack (~30ns vs ~1ns) |
| GC pressure | Nhiều heap = nhiều GC work |
| Benchmarking | Benchmarks có thể misleading nếu không escape |
| Security | Stack variables không bị accessed sau function return |

## Stack Growth Mechanism

Go dùng **contiguous stacks** (Go 1.4+) — khác với split stacks (Go 1.3 trở về trước):

```
Stack growth:
1. Bắt đầu ~2KB per goroutine
2. Khi call stack dùng hết space:
   a. Allocate buffer mới (2× size)
   b. Copy toàn bộ stack frames sang buffer mới
   c. Update tất cả stack pointers
3. Khi goroutine idle lâu: shrink (thu nhỏ lại)
```

### ⚠️ Stack Scanning in GC

- GC phải **scan tất cả goroutine stacks** để tìm pointers → GC overhead
- 100K goroutines × 8KB avg stack = **800MB** chỉ cho stack memory!

```go
// Monitor goroutine count
import "runtime"

fmt.Println("Goroutines:", runtime.NumGoroutine())

// Debug: dump all goroutine stacks
import "runtime/debug"
debug.Stack()                     // current goroutine only
runtime.Stack(buf, true)          // all goroutines
```

### Stack vs Heap — Decision Matrix

| Scenario | Stack | Heap |
|---------|-------|------|
| Local vars không escape | ✅ | — |
| Return pointer of local var | — | ✅ |
| Size unknown at runtime | — | ✅ |
| Shared between goroutines | — | ✅ |
| Interface boxing | — | ✅ (usually) |
| Large allocation (> 32KB) | — | ✅ |
| Goroutine lifetime exceeded | — | ✅ |

## Memory Alignment & Padding

Go aligns struct fields theo **hardware alignment rules** — ảnh hưởng đến memory footprint:

```go
// ❌ Wasteful layout  — 24 bytes
type Bad struct {
    a bool    // 1 byte + 7 padding
    b float64 // 8 bytes
    c bool    // 1 byte + 7 padding
}
// unsafe.Sizeof(Bad{}) = 24

// ✅ Optimal layout (sort by size desc) — 16 bytes
type Good struct {
    b float64 // 8 bytes
    a bool    // 1 byte
    c bool    // 1 byte + 6 padding
}
// unsafe.Sizeof(Good{}) = 16
```

### Kiểm tra và fix

```bash
go vet -fieldalignment ./...  # suggest optimal field order
```

```go
import "unsafe"
fmt.Println(unsafe.Sizeof(Bad{}))   // 24
fmt.Println(unsafe.Alignof(Bad{}))  // 8
fmt.Println(unsafe.Offsetof(Bad{}.b)) // 8 (after 7 bytes padding)
```

### Trade-off: Readability vs Memory

| Priority | Approach |
|----------|---------|
| Readability | Group fields logically (user → address → metadata) |
| Memory | Sort by size: largest first → least padding |
| High-frequency alloc (millions) | ✅ Optimize alignment → significant savings |
| Rare alloc | ❌ Not worth the complexity |

## GOMEMLIMIT (Go 1.19+)

```go
import "runtime/debug"

// Giới hạn tổng memory Go runtime có thể sử dụng
debug.SetMemoryLimit(512 * 1024 * 1024)  // 512MB

// Environment variable (recommended for containers)
// GOMEMLIMIT=512MiB go run main.go
```

### GOGC vs GOMEMLIMIT — Combined Strategy

| | GOGC | GOMEMLIMIT |
|---|---|---|
| Controls | GC trigger frequency | Max heap size |
| Unit | Percentage (%) | Absolute bytes |
| Purpose | Throughput tuning | OOM prevention |
| Default | 100 | MaxInt (unlimited) |

```
GOGC=100 + GOMEMLIMIT=512MiB:
  → GC triggers when heap doubles
  → OR when heap approaches 512MiB limit (whichever first)
  → Best of both: throughput + safety

Production recommendation:
  GOMEMLIMIT = 90% of container memory limit
```

## Reducing GC Pressure — sync.Pool

```go
// ✅ Object pooling: reuse objects thay vì allocate mới mỗi lần
var bufPool = sync.Pool{
    New: func() any { return new(bytes.Buffer) },
}

func encode(data any) []byte {
    buf := bufPool.Get().(*bytes.Buffer)
    defer func() {
        buf.Reset()
        bufPool.Put(buf)  // return for reuse
    }()
    json.NewEncoder(buf).Encode(data)
    return append([]byte{}, buf.Bytes()...)
}
```

### sync.Pool Trade-offs

| | sync.Pool | Manual cache | Pre-allocated slice |
|---|---|---|---|
| GC cleared | ✅ (auto) | ❌ (must manage) | ❌ |
| OOM risk | ✅ Safe | ⚠️ | ✅ |
| Performance | ✅ | ✅ | ✅✅ |
| Complexity | ✅ Low | ⚠️ High | ⚠️ Medium |
| Use case | Short-lived buffers | Session caches | Fixed-size batch |

> **⚠️ Quan trọng**: Pool bị **cleared mỗi GC cycle**. KHÔNG dùng pool cho connection pools hay long-lived objects.

## Finalizers (Advanced — Use with Caution)

```go
import "runtime"

type Resource struct{ fd int }

func NewResource() *Resource {
    r := &Resource{fd: 42}
    runtime.SetFinalizer(r, func(r *Resource) {
        syscall.Close(r.fd)  // cleanup khi GC thu hồi r
    })
    return r
}
```

### ⚠️ Finalizer Pitfalls

| Issue | Chi tiết |
|-------|---------|
| Non-deterministic | GC quyết định khi nào finalizer chạy |
| No guarantee | Finalizer có thể **không** chạy khi program exit |
| Resurrection | Finalizer có thể cho object "sống lại" |
| GC overhead | Mỗi finalizer = thêm 1 GC cycle để reclaim |
| Goroutine leak | Finalizer chạy trong goroutine riêng |

> **Best practice**: Dùng `defer f.Close()` và explicit cleanup. Finalizer chỉ là *safety net* cuối cùng, không thay thế explicit resource management.

## GC Tuning — Practical Guide

```
Chiến lược theo workload:

High-throughput, latency không quan trọng:
  GOGC=200-400 → ít GC cycles, dùng thêm memory

Low-memory (containers, edge):
  GOGC=50-80 + GOMEMLIMIT=<limit> → frequent GC, bounded memory

Batch processing:
  GOGC=off → disable auto GC
  runtime.GC() manually at checkpoints

Production default (recommended):
  GOGC=100 (default)
  GOMEMLIMIT=90% of container limit
```
