# 7.4 Service Mesh

## Overview

Service Mesh = **dedicated infrastructure layer** xử lý service-to-service communication. Tách biệt network logic (retry, mTLS, circuit breaking) khỏi application code.

```
┌─────────────────────────────────────────────────────────────────┐
│              WITHOUT Service Mesh                               │
│  ┌──────────────┐                                               │
│  │  Service A   │───► Application code phải tự implement:      │
│  │              │     - Retry logic                            │
│  │  App Code    │     - Circuit breaker                       │
│  │  + Network   │     - mTLS (certificate rotation)           │
│  │  + Retry     │     - Observability (metrics/traces)        │
│  │  + mTLS      │     → Coupling, duplicated effort           │
│  └──────────────┘                                               │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│              WITH Service Mesh                                  │
│  ┌──────────────┐     ┌──────────────┐                        │
│  │  Service A   │─────►│  Sidecar     │                         │
│  │              │     │  (Envoy)     │                         │
│  │  App Code    │     │  Proxy       │                         │
│  │  (clean)     │     │  ─────────── │                         │
│  └──────────────┘     │  mTLS        │                         │
│                        │  Retry       │                         │
│  ┌──────────────┐     │  Circuit Brk │                         │
│  │  Service B   │◄─────│  Metrics     │                         │
│  │              │     │  Tracing     │                         │
│  │  App Code    │     │  (offload)   │                         │
│  │  (clean)     │     └──────────────┘                        │
│  └──────────────┘                                               │
│                                                                  │
│  ┌────────────────────────────────────────────────────────────┐ │
│  │  Control Plane (Istiod / Linkerd control plane)           │ │
│  │  - Certificate authority (mTLS certificates)             │ │
│  │  - Traffic policies (retries, timeouts, circuit breaker)  │ │
│  │  - Telemetry collection                                    │ │
│  │  - Config下发 (xDS protocol)                              │ │
│  └────────────────────────────────────────────────────────────┘ │
└────────────────────────────────────────────────────────────────┘
```

## Sidecar Pattern

### Without Sidecar (Traditional)

```
┌─────────────────────────────────────────┐
│  Pod                                     │
│  ┌─────────────────────────────────┐    │
│  │  Container: myapp               │    │
│  │  ├─ App code                   │    │
│  │  └─ Network client             │    │
│  │     (rety, TLS, observability) │    │
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

### With Sidecar (Service Mesh)

```
┌─────────────────────────────────────────┐
│  Pod                                     │
│  ┌─────────────────────────────────┐    │
│  │  Container: myapp               │    │
│  │  Container: istio-proxy (sidecar)│   │
│  │  Container: prometheus-exporter │    │
│  └─────────────────────────────────┘    │
│                                          │
│  Container network: localhost            │
│  myapp → localhost:15001 → proxy → peer │
└─────────────────────────────────────────┘
```

## mTLS (Mutual TLS)

### How mTLS Works

```
┌──────────────────────────────────────────────────────────────┐
│  mTLS Handshake                                               │
│                                                               │
│  Service A ──────────────► Service B                          │
│      │                                                        │
│      │  ClientHello + A's certificate                        │
│      │─────────────────────────────────────────────────►     │
│      │                                                        │
│      │  ServerHello + B's certificate                         │
│      │  + certificate request (⚠️ mTLS asks for client cert)  │
│      │◄──────────────────────────────────────────────────     │
│      │                                                        │
│      │  ClientCertificate + B verifies A's cert               │
│      │  Key exchange (e.g., ECDHE)                            │
│      │─────────────────────────────────────────────────►     │
│      │                                                        │
│      │  A verifies B's cert + encrypted channel established   │
│      │◄──────────────────────────────────────────────────     │
│                                                               │
│  Result: BOTH sides authenticate each other                   │
└──────────────────────────────────────────────────────────────┘
```

### Service Mesh mTLS (Istio)

```yaml
# PeerAuthentication — enforce mTLS at namespace level
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: production
spec:
  mtls:
    mode: STRICT  # STRICT = only mTLS allowed, PERMISSIVE = both mTLS + plaintext

---
# DestinationRule — TLS settings per service
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: myapp-dr
spec:
  host: myapp.production.svc.cluster.local
  trafficPolicy:
    tls:
      mode: ISTIO_MUTUAL  # Use mesh's internal certificate
    portLevelSettings:
      - port:
          number: 8080
        tls:
          mode: ISTIO_MUTUAL
```

### Linkerd mTLS (Automatic)

Linkerd tự động enable mTLS cho tất cả traffic mà không cần config thêm — chỉ cần namespace được annotated.

```yaml
# Linkerd: enable mTLS cho namespace
apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    linkerd.io/inject: enabled
```

## Traffic Management

### VirtualService — Request Routing

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: myapp-vsvc
spec:
  hosts:
    - myapp.production.svc.cluster.local
  gateways:
    - myapp-gateway
  http:
    # Route: /api → v2
    - name: api-route
      match:
        - uri:
            prefix: /api
      route:
        - destination:
            host: myapp-v2.production.svc.cluster.local
          weight: 100

    # Route: / → v1
    - name: default-route
      match:
        - uri:
            prefix: /
      route:
        - destination:
            host: myapp-v1.production.svc.cluster.local
          weight: 100

    # Canary: 5% traffic to v2
    - name: canary-route
      match:
        - headers:
            x-canary:
              exact: "true"
      route:
        - destination:
            host: myapp-v2.production.svc.cluster.local
          weight: 100
```

### Weighted Routing (Canary)

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: myapp-canary
spec:
  hosts:
    - myapp.production.svc.cluster.local
  http:
    - route:
        - destination:
            host: myapp-stable.production.svc.cluster.local
          weight: 95
        - destination:
            host: myapp-canary.production.svc.cluster.local
          weight: 5
```

### Header-based Routing

```yaml
# Traffic: User-Agent based routing
spec:
  http:
    - match:
        - headers:
            user-agent:
              regex: ".*Mobile.*"
      route:
        - destination:
            host: myapp-mobile.production.svc.cluster.local
    - route:
        - destination:
            host: myapp.production.svc.cluster.local
```

## Traffic Split (Progressive Delivery)

### Argo Rollouts + Service Mesh

```yaml
# Rollout with Istio traffic split
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: myapp
spec:
  replicas: 5
  strategy:
    canary:
      trafficRouting:
        - istio:
            virtualService:
              name: myapp-vsvc
              routes:
                - primary  # VirtualService route name
      steps:
        - setWeight: 10    # 10% to canary
        - pause: {duration: 5m}
        - setWeight: 30
        - pause: {duration: 10m}
        - setWeight: 100
```

```yaml
# VirtualService updated by Argo Rollouts
# (Rollout controller dynamically updates weight)
spec:
  http:
    - route:
        - destination:
            host: myapp-stable
            weight: 90
        - destination:
            host: myapp-canary
            weight: 10
```

## Retry & Timeout

### Retry Policy

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: myapp-retry
spec:
  hosts:
    - myapp.production.svc.cluster.local
  http:
    - route:
        - destination:
            host: myapp.production.svc.cluster.local
      retries:
        attempts: 3        # Retry 3 lần
        perTryTimeout: 2s  # Mỗi attempt timeout 2s
        retryOn: 5xx,reset # Retry on 5xx errors, connection reset
        retryRemoteLocalities: true  # Retry across AZs on failure
      timeout: 10s
```

### Timeout Policy

```yaml
spec:
  http:
    - match:
        - uri:
            prefix: /api/sync
      route:
        - destination:
            host: myapp-api.production.svc.cluster.local
      timeout: 30s   # Total timeout

    - match:
        - uri:
            prefix: /api/async
      route:
        - destination:
            host: myapp-api.production.svc.cluster.local
      timeout: 5m    # Longer timeout for async operations
```

## Circuit Breaker

### Envoy Circuit Breaker (Istio)

```yaml
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: myapp-cb
spec:
  host: myapp.production.svc.cluster.local
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100
      http:
        h2UpgradePolicy: UPGRADE
        http1MaxRequestHeadersSize: 8192
    outlierDetection:
      consecutive5xxErrors: 5          # 5 lỗi 5xx liên tiếp
      interval: 10s                    # trong 10s
      baseEjectionTime: 30s            # eject host trong 30s
      maxEjectionPercent: 50            # tối đa 50% hosts bị eject
      minHealthPercent: 30              # disable CB khi >70% healthy
```

### Circuit Breaker States

```
┌───────────────────────────────────────────────────────────────┐
│  Circuit Breaker States                                       │
│                                                               │
│  CLOSED ──[5 consecutive 5xx]──► OPEN                       │
│    │  (normal traffic)          (all requests fail fast)      │
│    │                             │                             │
│    │◄──[timeout: 30s]───────────┘                             │
│    │                                                           │
│    ▼                                                           │
│  HALF-OPEN                                                     │
│    (1 test request)──►[success]──► CLOSED                     │
│                      └──►[fail]──────► OPEN                   │
└───────────────────────────────────────────────────────────────┘
```

### Connection Pool Settings

```yaml
spec:
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100           # Max HTTP/TCP connections to upstream
        connectTimeout: 10s           # TCP connection timeout
      http:
        http1MaxPendingRequests: 50  # Max pending HTTP requests
        http2MaxRequests: 1000        # Max requests for HTTP/2
        maxRequestsPerConnection: 100  # Max requests per HTTP/2 connection
        maxRetries: 3                  # Max retries
```

## Rate Limiting

### Local Rate Limit (per instance)

```yaml
# EnvoyFilter: rate limit per route
apiVersion: networking.istio.io/v1alpha3
kind: EnvoyFilter
metadata:
  name: myapp-ratelimit
  namespace: production
spec:
  workloadSelector:
    labels:
      app: myapp
  configPatches:
    - applyTo: VIRTUAL_HOST
      patch:
        operation: MERGE
        value:
          rate_limits:
            - actions:
                - header_value_match:
                    descriptor_value: "v1"
                    headers:
                      - name: ":path"
                        exact_match: /api/orders
              local_rate_limit:
                requests_per_unit_of_time: 100
                unit: minute
```

### Global Rate Limit (Redis-based)

```yaml
# Redis global rate limiting via Envoy
apiVersion: networking.istio.io/v1alpha3
kind: EnvoyFilter
metadata:
  name: myapp-global-ratelimit
spec:
  configPatches:
    - applyTo: CLUSTER
      patch:
        operation: MERGE
        value:
          lb_config:
            locality_lb_config: {}

    - applyTo: HTTP_FILTER
      match:
        listener:
          filter_chain:
            filter:
              name: envoy.filters.network.http_connection_manager
      patch:
        operation: INSERT_BEFORE
        value:
          name: envoy.ratelimit
          typed_config:
            "@type": type.googleapis.com/envoy.extensions.ratelimit.dynamic.v3.RateLimitServiceV3.RateLimitServiceConfig
            grpc_service:
              envoy_grpc:
                cluster_name: rate_limit_cluster
            transport_api_version: V3
```

## Security (Authorization)

### AuthorizationPolicy

```yaml
# Default: deny all
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: myapp-authz
  namespace: production
spec:
  selector:
    matchLabels:
      app: myapp
  rules:
    # Frontend → API: only allow GET and POST
    - from:
        - source:
            principals:
              - cluster.local/ns/production/sa/frontend
      to:
        - operation:
            methods: ["GET", "POST"]
            paths: ["/api/*"]
    # Admin → All endpoints
    - from:
        - source:
            principals:
              - cluster.local/ns/production/sa/admin
      to:
        - operation:
            methods: ["*"]
            paths: ["*"]
    # Internal services only
    - from:
        - source:
            namespaces: ["production"]
```

### L7 (Application Layer) Authorization

```yaml
# Istio Authorization (L7) — inspect HTTP headers/paths
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: api-authz
spec:
  selector:
    matchLabels:
      app: myapp-api
  rules:
    - from:
        - source:
            principals: ["*"]
      to:
        - operation:
            methods: ["GET"]
            paths: ["/api/users"]
      conditions:
        - key: request.headers[X-User-Role]
          values: ["admin", "user"]
    - from:
        - source:
            principals: ["*"]
      to:
        - operation:
            methods: ["DELETE"]
            paths: ["/api/users/*"]
      conditions:
        - key: request.headers[X-User-Role]
          values: ["admin"]
```

## Observability Integration

### Automatic Metrics

```yaml
# Enable telemetry v2 (automatic mTLS metrics, TCP metrics)
apiVersion: telemetry.istio.io/v1alpha1
kind: Telemetry
metadata:
  name: myapp-telemetry
  namespace: production
spec:
  tracing:
    - providers:
        - name: jaeger
      randomSamplingPercentage: 10.0
  metrics:
    - providers:
        - name: prometheus
      overrides:
        - matchModes:
            - mode: CLIENT_AND_SERVER
          metrics:
            - requests_total
            - request_duration_milliseconds
            - responses_total
        - matchModes:
            - mode: SERVER
          disabled: false
```

### Access Logging

```yaml
apiVersion: telemetry.istio.io/v1alpha1

  name: myapp-logging
spec:
  logging:
    providers:
      - name: stackdriver
    accessLogging:
      - providers:
          - name: envoy
        disabled: false
```

## Linkerd vs Istio

| Feature | Linkerd | Istio |
|---------|--------|-------|
| **Complexity** | Simple, minimal | Complex, feature-rich |
| **mTLS** | Auto (zero-config) | Requires config |
| **Performance** | Lower overhead | Higher overhead |
| **Protocols** | HTTP, gRPC, TCP | HTTP, gRPC, TCP, WebSocket |
| **Traffic mgmt** | Basic | Full (header, weight, mirror) |
| **Extensibility** | Limited | Very high (EnvoyFilters) |
| **Learning curve** | Low | High |
| **Best for** | Simple microservices | Complex mesh needs |
| **CNCF** | ✅ Incubating | ✅ Graduated |

### When to Use Linkerd

- Simple HTTP/gRPC microservices
- Team nhỏ, cần security (mTLS) nhanh
- K8s-first, không cần complex routing
- Want simplicity over features

### When to Use Istio

- Complex routing (header, weighted, mirroring)
- Multiple protocols (TCP, gRPC, WebSocket)
- Fine-grained authorization policies
- Deep observability needs
- Multi-cluster federation

## Sidecar Injection

### Automatic Injection

```bash
# Istio: label namespace for auto-injection
kubectl label namespace production istio-injection=enabled

# Tất cả pods mới trong namespace sẽ có sidecar tự động

kubectl rollout restart deployment/myapp -n production
```

### Manual Sidecar (exclude specific pods)

```yaml
# Exclude sidecar cho specific pod
metadata:
  annotations:
    sidecar.istio.io/inject: "false"
```

## 📌 Tóm tắt

```
Service Mesh
│
├── Architecture
│   ├── Sidecar Pattern ── Envoy proxy per pod (localhost)
│   └── Control Plane ─── Certificate authority, config push (xDS)
│
├── mTLS
│   ├── STRICT mode ────── Only mTLS traffic (no plaintext)
│   ├── PERMISSIVE ─────── Both mTLS + plaintext
│   └── ISTIO_MUTUAL ──── Use mesh-managed certificates
│
├── Traffic Management
│   ├── VirtualService ──── HTTP routing (header, weight, path)
│   ├── DestinationRule ─── TLS, connection pool, circuit breaker
│   └── Gateway ─────────── Ingress/Egress configuration
│
├── Resilience
│   ├── Retry ───────────── attempts, perTryTimeout, retryOn
│   ├── Timeout ─────────── request-level timeouts
│   ├── Circuit Breaker ── consecutive5xx → eject host
│   └── Rate Limit ──────── Local (per instance) or Global (Redis)
│
├── Security
│   ├── AuthorizationPolicy ──── L3/L4/L7 access control
│   └── PeerAuthentication ────── mTLS enforcement
│
├── Observability (automatic)
│   ├── Metrics ────────────── request_total, duration, TCP metrics
│   ├── Traces ────────────── OpenTelemetry, Jaeger
│   └── Access Logs ───────── Envoy access logging
│
└── Tools Comparison
    ├── Linkerd ────────────── Simple, auto-mTLS, low overhead
    └── Istio ─────────────── Complex, full features, high overhead
```
