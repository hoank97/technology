## Kafka

### Đặc điểm

| | |
|---|---|
| **Model** | Pull-based |
| **Message model** | Distributed log (append-only) |
| **Message deletion** | Retained (configurable) |
| **Consumer** | Consumer group, tự quản offset |
| **Use case** | Event streaming, CDC |

### Kiến trúc

```
Producer → Topic (Partition) → Consumer Group A/B/C
                              ↑          ↑
                         Each group gets all messages
```

### Delivery Semantics

| | At-most once | At-least once | Exactly once |
|---|---|---|---|
| Description | Có thể miss | Có thể duplicate | Đúng 1 lần |
| Commit offset | Trước khi xử lý | Sau khi xử lý | Transaction |
| Use case | Low priority | OK to retry | Billing |

## RabbitMQ vs Kafka

| | RabbitMQ | Kafka |
|---|---|---|
| Model | Push | Pull |
| Message retention | Deleted after consume | Retained (disk) |
| Replay | ❌ Không | ✅ Có |
| Throughput | ~50K msg/s | ~1M+ msg/s |
| State tracking | Broker track | Consumer track offset |
| Routing | Flexible (exchange) | Topic + partition key |

## Edge Cases

### Lost Event

**Process thành công nhưng MQ publish fail → Event bị mất.**

**Solution: Outbox Pattern** (xem [[7.3 - Transaction & Saga]])

### Duplicate Event

**Event được publish nhiều lần → Duplicate processing.**

**Solution: Idempotency**

## Idempotency Patterns

### 1. Idempotency Key

```go
type IdempotentHandler struct {
    seen sync.Map
}

func (h *IdempotentHandler) Handle(msg Message) error {
    if _, exists := h.seen.Load(msg.ID); exists {
        return nil // Skip, đã xử lý
    }

    err := process(msg)
    if err == nil {
        h.seen.Store(msg.ID, true)
    }
    return err
}
```

### 2. Database Upsert

```sql
INSERT INTO payments (id, amount, status)
VALUES ($1, $2, $3)
ON CONFLICT (id) DO NOTHING
```

### 3. Natural Idempotency

```sql
UPDATE accounts SET status = 'completed' WHERE id = $1
-- Thực hiện 2 lần → vẫn = 'completed'
```

## Race Condition

**2+ operations cùng chạy, kết quả phụ thuộc vào timing.**

### Ví dụ

```go
// 2 requests cùng đọc balance = 100
// Cả 2 đều write balance = 100 + 50 = 150
// Thực tế nên = 200

Request A: Read balance → 100
Request B: Read balance → 100
Request A: Write balance → 100 + 50 = 150
Request B: Write balance → 100 + 50 = 150  ← WRONG!
```

### Solutions

```go
// 1. Optimistic Locking
UPDATE accounts SET balance = balance + 50, version = version + 1
WHERE id = $1 AND version = $2

// 2. Pessimistic Locking
SELECT * FROM accounts WHERE id = $1 FOR UPDATE
```
