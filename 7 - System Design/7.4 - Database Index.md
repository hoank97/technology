## Index Principles

### Nên đánh index cho columns:

| | |
|---|---|
| **WHERE clauses** | `WHERE email = '...'` |
| **JOIN columns** | `JOIN ON user_id = id` |
| **ORDER BY** | `ORDER BY created_at` |
| **UNIQUE constraints** | email phải unique |

### Composite Index (Multi-column)

```sql
CREATE INDEX idx_user ON orders(user_id, created_at DESC);
```

**Column order matters!**

```
Query: WHERE user_id = 5 AND created_at > '2024-01-01'
→ ✅ Index used: (user_id, created_at)

Query: WHERE created_at > '2024-01-01'
→ ❌ Index NOT used: (user_id, created_at) — wrong order
```

→ **Leftmost prefix rule**: composite index dùng được cho prefix columns

## Trade-offs

| | Index có lợi | Index có hại |
|---|---|---|
| **Read** | ✅ Nhanh hơn | |
| **INSERT/UPDATE/DELETE** | | ❌ Chậm hơn (phải update index) |
| **Storage** | | ❌ Tốn thêm disk space |

### Quá nhiều index

- ❌ Chậm INSERT/UPDATE/DELETE
- ❌ Tốn storage
- ❌ Query planner có thể **confused**

---

## Types of Index

| Type | Mô tả |
|------|--------|
| **B-Tree** (default) | Cân bằng tree, tốt cho range queries |
| **Hash** | O(1) lookup, không dùng cho range |
| **Full-text** | Text search |
| **Composite** | Nhiều columns |

---

## Best Practices

```sql
-- ✅ Tốt: Index cho WHERE clause thường dùng
CREATE INDEX idx_user_email ON users(email);

-- ✅ Tốt: Composite index cho query pattern cụ thể
CREATE INDEX idx_orders_user_date ON orders(user_id, created_at DESC);

-- ❌ Bad: Index cho column ít dùng trong WHERE
CREATE INDEX idx_user_age ON users(age);

-- ❌ Bad: Quá nhiều indexes
--   → 5+ indexes trên table nhỏ thường overkill
```
