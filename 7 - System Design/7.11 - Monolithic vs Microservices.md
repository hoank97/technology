## Microservices

### Advantages

✅ Resolve all disadvantages of Monolithic:

| | |
|---|---|
| ✅ Scale independently | Scale order-service without scaling user-service |
| ✅ Independent deployment | Fix bug in service A doesn't affect service B |
| ✅ Tech stack flexibility | Order: Go, Payment: Python, Analytics: Rust |
| ✅ Team autonomy | Each team owns their service |
| ✅ Fault isolation | Payment down doesn't break entire app |

### Disadvantages

| | |
|---|---|
| ❌ Network latency | Function call (ns) → HTTP call (ms) |
| ❌ Distributed debugging | Stack traces are no longer enough |
| ❌ Data consistency | No more simple database transactions |
| ❌ Operational overhead | More services = more to deploy, monitor, maintain |
| ❌ Testing complexity | Integration tests are much harder |

---

## So sánh

| | Monolithic | Microservices |
|---|---|---|
| Deployment | Single artifact | Independent per service |
| Scaling | Scale all | Scale individual |
| Coupling | Tight | Loose |
| Debug | Easy (single process) | Hard (distributed) |
| Data consistency | ACID | Eventual (Saga) |
| Team size | Small (<10) | Large (>20) |
| Complexity | Low | High |

---

## When do you need Microservices?

| | |
|---|---|
| ✅ | Multiple teams need to deploy independently |
| ✅ | Different parts have different scaling needs |
| ✅ | Domain boundaries are clear |
| ✅ | Have DevOps capability (CI/CD, monitoring, etc.) |
| ❌ | Small team, simple domain → Monolithic first |

## Migration: Strangler Fig Pattern

```
Từ từ thay thế monolith → microservices

1. Deploy proxy trước monolith
2. Redirect traffic nhỏ sang service mới
3. Mỗi feature hoàn thành → redirect thêm
4. Khi monolith không còn traffic → tắt đi
```

## Structured Monolith (Middle Ground)

Vẫn monolith nhưng có **module boundaries rõ ràng**:

```
├── internal/
│   ├── orders/       # Module riêng
│   ├── payments/     # Module riêng
│   └── users/        # Module riêng
├── main.go
```

→ **Deploy cùng nhau** nhưng **boundaries rõ** → dễ tách ra sau
